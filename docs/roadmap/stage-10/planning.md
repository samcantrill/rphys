## Roadmap Extraction

- Baseline roadmap outcome: define the boundary between generic `Batch` containers, batch-level prediction or representation `Method` objects, and lower-level `Model` computation without implementing concrete algorithms.
- Prerequisites: Stage 1 field/metadata vocabulary; Stage 2 `Sample`/`Batch` containers and collation; a new Phase 0 repair for LIST uncollation and generic metadata/provenance preservation; Stage 6/7 operation/mutation vocabulary; Stage 8 export keeps durable writes out of methods; landed Stage 9 framework iteration and `BatchCollater` code provide an integration path but remaining Stage 9 work is not required for base method/model unit tests.
- Prior or adjacent roadmap links: Stage 9 supplies `SampleSource`/torch-loader/batch-collater context; Stage 11 consumes method predictions for losses/objectives/metrics; Stage 12 composes `Method` inside learners/trainers; Stage 13 treats predictions and representations as normal fields for export/evaluation/analysis.
- Primary feature docs: none found; `docs/roadmap.md` is canonical.
- Deferred or out-of-scope roadmap work: concrete CHROM/POS methods, neural baselines, model zoo, concrete tensor kernels, concrete loss/objective/metric catalog, learners/trainers, optimizers/schedulers/checkpoints, device/distributed behavior, dataloaders, prediction export, evaluation reports, framework runtimes, Lightning/Fabric adapters, and project configuration systems.
- Compatibility obligations: public imports remain scoped and lightweight; `Method.predict` is loss-free/objective-free and does not export, train, step optimizers, update metrics, choose splits, or write checkpoints; field locators and metadata selectors are parsed before hot loops; trainable capability protocols do not import torch in core modules; models do not depend on datasource/index/sample-builder/dataloader objects.
- Public surfaces or durable artifacts likely affected: likely code-backed public surfaces under `rphys.methods` for `Method`, `StatefulMethod`, `TrainableMethod`, adapters, `MethodOutput`, and `PredictionContext`; under `rphys.models` for `Model` and model input/result records or protocols; under `rphys.nn` for dependency-light protocol helpers only if design proves they are needed. No durable file artifacts are expected in this stage.

## Overview

- What this stage covers: base method/model contracts, typed input/output adapters over `Batch` fields, prediction context, method output records, explicit state/parameter capability protocols, method/model failure behavior, and lightweight public imports/tests/docs.
- Why it exists: later learners, trainers, losses, metrics, prediction export, and synthetic smoke tests need a stable boundary where batch-level prediction semantics are separate from lower-level model execution and from optimization/training mechanics.
- Primary outcomes: a base `Method` can adapt declared `Batch` fields to structured model or operation inputs, call a lower-level model/callable when configured, and return prediction or representation fields under declared `FieldLocator`s; lower-level models are testable without runtime datasource machinery; stateful/trainable capabilities expose state/parameters without owning optimizers, devices, checkpoints, or distributed behavior.
- Non-goals: concrete rPPG algorithms, neural baselines, model zoo, loss/objective/metric behavior, learner/trainer loops, optimizer/scheduler/checkpoint/device policy, dataloader construction, export/materialization, evaluation/reporting, and heavy optional dependency installation.
- Related feature docs: none found.
- Impacted repo areas: pre-phase data repair in `src/rphys/data/collation.py`, `src/rphys/data/__init__.py`, and collation/collater tests; then likely `src/rphys/methods/`, `src/rphys/models/`, `src/rphys/nn/`, `src/rphys/errors.py` only for exercised method/model errors if needed, package import tests, method/model contract tests, unit tests, and synthetic support fixtures.
- Current state: package homes exist but export no public names; broad `RemotePhysMethodError` exists; runtime `Batch`/`FieldLocator` containers and `predictions`/`outputs` roles are available; no method/model/nn contract implementation or tests exist yet.
- Key uncertainty: exact public module split and class/protocol shapes are design-gate work. Functionality defaults strongly support protocol-only core trainable contracts with no torch hard dependency; optional torch-aware helpers should be import-gated or deferred unless design evidence proves an immediate need.
- User clarification questions and resolved answers: maintainer input for this run asks for extra attention to modular interfaces and for explicit surfacing of the torch dependency posture. This pass records the repo-resolved default as protocol-only core contracts, no hard torch dependency, and optional/import-gated torch integration only as a design-gate or later-stage candidate.
- Planning priority or optimization target: establish a small, modular, dependency-light contract layer that later stages can compose without forcing data loading, device movement, optimization, or framework-specific behavior into methods/models.

## Behavior Confirmation

- Included behavior: locked. Phase 0 LIST uncollation and generic metadata/provenance round trip; base method prediction contract, input/output adapters, `MethodOutput`, `PredictionContext`, lower-level `Model`, stateful/trainable capability protocols, dependency-light public exports, typed failures, and synthetic contract tests.
- Default behavior: methods consume `Batch` fields through construction-time parsed selectors/adapters, return prediction or representation fields, and do not mutate input batches unless declared. Models consume structured inputs produced by adapters. Trainable/stateful capabilities are protocol-only and do not import torch.
- Failure behavior: invalid selectors, missing fields, wrong schema/type, invalid model output shape, unsupported mutation, undeclared output locators, invalid context, and unavailable optional dependencies fail loudly with typed errors or existing field/data errors.
- Unsupported behavior: concrete algorithms, method-owned losses/objectives/metrics, optimizer/scheduler/checkpoint/device/distributed behavior, datasource scanning, dataloader building, export/file writes, workflow runtime, hard torch dependency, and root exports.
- Resume/interruption behavior: planning workflow only; Stage 10 does not define workflow resume or trainer checkpoint behavior.
- Downstream implications: Stage 11 can consume method outputs for losses/metrics; Stage 12 can compose methods in learners/trainers and recover sample-level batch outputs through the data-layer inverse; Stage 13 can export/evaluate predictions as normal fields while adding any required domain-specific attribution outside Stage 10; Stage 15 can profile training without method contracts owning profiling.
- Explicit deferrals: exact API signatures, module split, public/protocol class choice, optional torch helper module shape, state-dict naming, structured model input/output record shape, method-output merge semantics, and specific error names are design-gate work.
- Why this behavior is locked: FQ-0 through FQ-8 are repo-resolved by maintainer direction, the roadmap, current runtime contracts, empty runtime dependency set, import-boundary tests, and adjacent stage boundaries. No high-impact functionality choice requires maintainer discussion before implementation.

## Proposed Implementation Shape

- Likely modules or packages:
  - `rphys.data.collation`: Phase 0 adds public `uncollate_batch(batch, ...) -> tuple[Sample, ...]` as the explicit inverse for default LIST collation. It should preserve FieldLocator keys, per-sample payloads, schema, field metadata presence, generic metadata/provenance, and collate policy where the batch is reversible, and fail loudly when sample count or field alignment is ambiguous.
  - `rphys.ops.batch`: no Stage 10-owned identity changes are planned. Existing operation contexts remain compatible, but Phase 0 should not add sample IDs, batch IDs, or other domain-specific metadata fields.
  - `rphys.methods.core`: `Method`, `StatefulMethod`, and `TrainableMethod` structural protocols. `Method.predict(batch, *, context=None) -> MethodOutput` is the semantic entrypoint; it is not `__call__`-only and not an operation pipeline step.
  - `rphys.methods.adapters`: `MethodInputSpec`, `MethodOutputSpec`, `MethodInputAdapter`, and `MethodOutputAdapter`. Specs are frozen records that normalize `FieldLocator`/`MetadataKey` values at construction and declare expected schema/payload type where needed. Adapters extract named model inputs from `Batch` and map named model outputs back to field locators.
  - `rphys.methods.context`: `PredictionContext`, a frozen dependency-light record with primitive generic `metadata` and `provenance` mappings only. Stage 10 should not define first-class sample IDs, batch IDs, run identifiers, split/mode labels, dtype/device fields, or other domain/workflow-specific context keys; downstream stages may carry such values in their own context adapters when they define the semantics.
  - `rphys.methods.output`: `MethodOutput`, a frozen patch-like record containing `fields: Mapping[FieldLocator, FieldValue]`, `diagnostics`, `metadata`, and `provenance`. It does not own export, metric aggregation, loss computation, or checkpointing. Merge/apply behavior, if included, should be an explicit helper that returns a new or chosen-copy `Batch`; `predict` should not silently mutate the input batch.
  - `rphys.methods.state`: richer backend-neutral state and trainable-handle records such as `StateEntry`, `StateView`, `StateLoadResult`, and `ParameterView`. These records carry names, values or handles, trainability flags where relevant, primitive metadata/provenance, and strict-load diagnostics without defining optimizer groups, checkpoint files, device movement, distributed state, or torch parameter semantics.
  - `rphys.models.core`: `Model` structural protocol over a generic input and output type, likely `__call__(inputs) -> outputs`. The core model contract does not import `Batch`, `PredictionContext`, datasources, loaders, torch, training, export, loss, or metric modules.
  - `rphys.nn.protocols`: backend-neutral protocol helpers only if they are used by the code-backed `StatefulMethod`/`TrainableMethod` surface. No `rphys.nn.torch` or torch import is proposed for Stage 10.
  - `rphys.errors`: keep existing `RemotePhysMethodError` and field errors as the default failure surface; add specific method/model errors only when tests prove a broad error would hide important context.
- Likely public classes/functions/protocols:
  - `uncollate_batch`
  - `Method`, `StatefulMethod`, `TrainableMethod`
  - `MethodInputSpec`, `MethodOutputSpec`
  - `MethodInputAdapter`, `MethodOutputAdapter`, `MethodOutput`
  - `PredictionContext`
  - `StateEntry`, `StateView`, `StateLoadResult`, `ParameterView`
  - `Model`
  - Possibly reusable protocol helpers such as `Stateful` and `Trainable` under `rphys.nn` only if they directly back `StatefulMethod`/`TrainableMethod`; otherwise keep `rphys.nn.__all__` empty for Stage 10.
- Likely internal helpers:
  - Locator/spec coercion, field-role validation, adapter uniqueness checks, field extraction, schema/type checks using `Batch.field()`/`require()`, model-output arity checks, `FieldValue` wrapping, patch merge helpers, context coercion, and execution-error wrapping.
  - Private helpers should live beside the public module they support, for example `rphys.methods._validation`, rather than in a new shared utility package.
- Data flow:
  0. Phase 0 guarantees default LIST batches can be unlisted back to `tuple[Sample, ...]` when all fields are aligned. Generic metadata/provenance is preserved where reversible; domain-specific identifiers are not inferred from arbitrary payloads or added as first-class Stage 10 fields.
  1. Downstream code constructs input/output specs or adapters with typed or string locators. Strings are parsed immediately into `FieldLocator`s.
  2. Caller invokes `method.predict(batch, context=PredictionContext(...))`. The context carries only generic primitive metadata/provenance; any domain-specific identifiers remain caller-owned metadata outside the Stage 10 field contract.
  3. `MethodInputAdapter` reads declared `Batch` fields and builds a backend-neutral named mapping or caller-specified structured object for the model.
  4. The method calls a lower-level `Model` or importable callable if it uses one. Classical methods may compute directly without a model object.
  5. `MethodOutputAdapter` maps named model/method outputs to declared prediction, output, or diagnostic `FieldLocator`s and wraps payloads in `FieldValue` where needed.
  6. `MethodOutput` returns a patch-like field mapping plus diagnostics/provenance. Callers, learners, prediction helpers, or explicit merge helpers decide whether to attach fields to a `Batch`.
  7. Stateful/trainable methods expose state through `StateView`/`StateEntry` records and trainable handles through `ParameterView` records. These records are inspectable by later learners/trainers but do not move devices, group optimizer parameters, write checkpoints, or imply torch.
- Dependency direction:
  - `rphys.methods` may import `rphys.data` container/field/locator primitives, `rphys.models` protocol types, and optional backend-neutral `rphys.nn` protocols. It must not import datasources, Stage 9 torch loaders, losses, objectives, metrics, learners, trainers, prediction runners, export/materialization, workflow code, or torch.
  - `rphys.models` may import only Python typing/protocol utilities and light local validation. It should not import `Batch`, `Sample`, `FieldLocator`, datasources, loaders, methods, torch, training, export, losses, objectives, or metrics.
  - `rphys.nn` remains protocol-only and import-light. If later torch helpers are approved, they must be import-gated in an optional submodule and absent from base package imports.
- Extension points:
  - Downstream projects implement importable `Method`, `Model`, adapters, and specs by `_target_` paths. No registry is proposed.
  - Method adapters are the extension boundary for second datasets, second modalities, or project-specific model input conventions; downstream users should extend adapter/spec classes or pass importable adapter objects rather than edit method internals.
  - Backend-specific trainability, devices, optimizers, schedulers, checkpoint formats, compiled kernels, and framework integration remain additive future adapters owned by Stage 12+ or optional backend modules.
- Compatibility constraints:
  - Root `rphys.__all__` stays empty.
  - `rphys.methods`, `rphys.models`, and `rphys.nn` expose only code-backed names through scoped package/module `__all__` tests.
  - Core Stage 10 imports must not load torch, numpy, scipy, pandas, OpenCV, PyAV, matplotlib, Lightning, Fabric, dataset SDKs, or test support.
  - No placeholder modules, registries, fake model zoo entries, concrete algorithms, root exports, durable file schemas, or workflow/runtime concepts are introduced.

## Key Assumptions, Risks, And Open Questions

### Assumptions

- Stage 10 can satisfy its roadmap definition of done with dependency-light Python protocols/classes and synthetic fake methods/models.
- `Batch` remains the method input container; Stage 10 does not need a new model-specific runtime container.
- FieldLocator-keyed patch outputs are locked for Stage 10; later loss/metric/export/evaluation stages may convert or merge them explicitly when they need `Batch` objects.
- Core trainable/stateful capability protocols should describe state and parameter access through richer backend-neutral records without assuming torch tensors or torch `nn.Module`.
- No Stage 10 optional torch helper is required to satisfy the locked behavior baseline.
- Private/local validation helpers are acceptable for Stage 10 and should be refactored only after repeated public surfaces need identical semantics.
- Default LIST collation should be reversible through the data layer before Stage 10 method phases rely on sample reconstruction.

### Risks

- Public API shape can become too broad if `Method`, `Model`, adapters, state, and NN helpers are collapsed into one base class.
- A hard torch dependency would violate current import-boundary tests and make base imports heavy.
- Too little structure in `MethodOutput` could force Stage 11-13 to invent incompatible prediction/result shapes.
- Returning a full output `Batch` from `Method.predict` would be convenient but could blur mutation and downstream export/training responsibilities.
- Too much structure in `Model` inputs could overfit one tensor backend or one downstream project.
- Lifecycle-style `Model` hooks would recreate torch/Lightning responsibilities in rphys and make real framework adapters harder to add; Stage 10 must stay at callable computation plus descriptive state/parameter views only.
- Richer state/parameter records add Stage 10 public surface and validation cost, and may still need additive fields when Stage 12 defines optimizer groups and checkpoint metadata.
- Stage 9 Phases 1-4 are landed, but Stage 10 integration examples must avoid depending on remaining Stage 9 materialization/batch-planning/data-path closeout surfaces.
- Roadmap shorthand says a trivial method returns a prediction `Batch`, but Stage 10 has locked patch-like `MethodOutput`; implementation must bridge the shorthand with explicit merge/apply behavior rather than changing `predict` to return `Batch`.
- Phase 0 adds public data-layer surface and must avoid representing private collation evidence as ordinary user metadata.

### Open Questions

- None after DD-0 design pass, model lifecycle guardrail review, validation, phase shaping, and plan quality gate. Reopen only if implementation cannot define or satisfy acceptance criteria for LIST uncollation round trip, accepted patch-only output, rich backend-neutral state/parameter records, no Stage 10 torch helper, no torch/Lightning-style model lifecycle hooks, model-below-`Batch` boundary, or trainer/export/loss/device/checkpoint deferrals.
