## Roadmap Extraction

- Baseline roadmap outcome: bridge `DataSourceIndex`, cache-backed samples, derived datasource indexes, and prepared materialized data to framework iteration through one `SampleSource` contract while preserving `FieldLocator`-keyed `Sample` behavior.
- Prerequisites: Stage 3 lazy refs and index items; Stage 4 codecs and `SampleBuilder`; Stage 5 datasource indexes and manifests; Stage 7 sample/batch operation contracts; Stage 8 export-derived datasource outputs for the derived-index branch.
- Prior or adjacent roadmap links: Stage 5 provides `DataSourceIndex` and sidecar entry provenance; Stage 4 provides lazy/eager sample construction; Stage 8 provides future derived datasource refs; Stage 10+ consume loaded batches without owning data loading; Stage 15 later hardens performance profiling and data-path optimization.
- Primary feature docs: none found; `docs/roadmap.md` is canonical.
- Deferred or out-of-scope roadmap work: concrete `LitData`, WebDataset, Zarr, Arrow, Parquet, or vendor backend adapters; GPU/device movement; model input tuple formatting; trainer/learner loops; prediction/evaluation/analysis reports; datasource scanning/split construction; export/save operation implementation; generic artifact/workflow runtime; stable DDP cache behavior before rank-safe tests.
- Compatibility obligations: keep imports lightweight; preserve `FieldLocator` keys and per-item metadata; do not let Torch datasets scan raw directories, build indexes, export fields, or format model inputs; fail loudly for missing fields, unsupported eager/lazy modes, and unproven deterministic equivalence; make cache keys and invalidation inputs explicit; keep prepared data immutable by version; keep optimized-storage backends replaceable and import-gated.
- Public surfaces or durable artifacts likely affected: datasource-facing modules for `SampleRequest`, `SampleSource`, `IndexSampleSource`, `CachedSampleSource`, `PreparedSampleSource`, cache records, prepared/materialization manifests, batch planning records, and an import-gated `rphys.datasources.torch` adapter module; distinct `DerivedIndexSampleSource`, real cache payload writers, and concrete optimized backend adapters are deferred behind recorded revisit triggers.

## Overview

- What this stage covers: framework-neutral sample-source contracts, index-backed sample access, optional torch-compatible dataset/data-loader adapters, FieldLocator-aware batch collation, deterministic worker/runtime context, explicit local cache primitives, prepared-data equivalence manifests, storage-neutral materialization/layout records, batch planning/cost metadata, and explicit loader/profile/benchmark records.
- Why it exists: training and evaluation loops need a common way to iterate `Sample` and `Batch` objects from indexes, caches, derived data, or prepared data without reintroducing hidden datasource scanning, preprocessing, model formatting, or workflow orchestration.
- Primary outcomes: `IndexSampleSource` returns the same FieldLocator-keyed `Sample` shape as `SampleBuilder`; Torch-compatible adapters consume `SampleSource`; caches are explicit, deterministic, invalidation-aware, and atomic locally; prepared/materialized data proves equivalence to a `SampleRequest`; batch planning can use precomputed cost/layout metadata.
- Non-goals: raw datasource discovery, split/group/index construction, export/save operations, concrete optimized-storage adapters, model/learner/trainer/device behavior, prediction and metric reporting, hidden preprocessing in `__getitem__`, generic workflow/artifact runtime, and stable distributed cache semantics before local semantics and rank-safe tests exist.
- Related feature docs: none found.
- Impacted repo areas: likely `src/rphys/datasources/` for sample-source, cache, prepared/materialization, and torch-adapter modules; `src/rphys/data/collation.py` by composition only; package import tests; datasource/sample-builder/cache/torch contract tests; synthetic integration tests.
- Current state: indexes, sample builders, LIST collation, and sample/batch operation contracts exist; no Stage 9 modules or tests exist; Stage 8 export-derived datasource implementation is planned but absent in this checkout.
- Key uncertainty resolved by design agreement: module split, source access API, cache staging, durable prepared manifest scope, and public provisional reader boundary are locked. Implementation planning must still preserve deferrals for distinct derived-source validation, real cache payload writers, concrete storage backends, active streaming/resume, and stable DDP cache coordination.
- User clarification questions and resolved answers: user clarified that `v9` means `docs/roadmap/stage-9/` and Milestone 9 around line 1284, not literal `## 9. Milestone 1`.
- Planning priority or optimization target: establish the smallest dependency-light path from `DataSourceIndex` to `SampleSource` to FieldLocator-preserving batches, then add explicit cache/prepared/materialization contracts without hiding scientific transformations or making torch/storage backends mandatory dependencies.

## Behavior Confirmation

- Included behavior: Stage 9 defines `SampleSource` request/iteration, `IndexSampleSource`, derived-index consumption through the normal source path, deterministic runtime/worker context evidence, import-gated torch adapters, FieldLocator-aware batch collation, explicit local cache primitives and `CachedSampleSource`, prepared-data equivalence and `PreparedSampleSource`, storage-neutral materialization/layout records, batch planning/cost metadata, and data-path evidence records. A distinct `DerivedIndexSampleSource` public class is deferred until it can add code-backed derived provenance validation.
- Default behavior: data loading is fail-loud, explicit, dependency-light, and FieldLocator-preserving. `IndexSampleSource` composes existing `DataSourceIndex` and `SampleBuilder`; torch/storage integrations remain optional/import-gated; collation delegates to generic `Batch`/`CollatePolicy`; caches and prepared data are opt-in and must prove request equivalence.
- Failure behavior: missing requested fields, unsupported eager/lazy mode, unproven prepared/cache equivalence, stale cache key or invalidation mismatch, corrupt/incomplete cache entries, unsupported optional dependencies, invalid materialization metadata, and unsupported distributed/streaming behavior fail loudly with typed diagnostics or typed result evidence.
- Unsupported behavior: torch as a base dependency, raw directory scanning inside dataset adapters, split/index construction, export/save operations, model tuple formatting, trainer/device movement, generic artifact/workflow runtime, concrete optimized backend adapters, implicit caches, hidden preprocessing in `__getitem__`, and stable DDP cache behavior without rank-safe tests.
- Resume/interruption behavior: Stage 9 may record `DataLoaderState` or streaming plan data as explicit contracts; it does not implement a generic workflow resume engine or hidden dataset/cache resume behavior.
- Downstream implications: Stage 9 unlocks Stage 10+ method/training consumption and Stage 15 data-path profiling by providing a common sample-source, cache, prepared-data, and batch-planning boundary that downstream layers can consume without owning datasource discovery or model formatting.
- Explicit deferrals: concrete optimized-storage adapters/dependencies, active distributed cache coordination, stable DDP cache semantics, model formatting, trainer loops, advanced streaming runtime, remote cache stores, cache eviction sophistication, and generic workflow/artifact behavior.
- Why this behavior is locked: FQ-1 through FQ-10 are repo-resolved and FQ-11 is repo-resolved/deferred from Milestone 9 text, existing `DataSourceIndex`/`SampleBuilder`/collation behavior, package import policy, and adjacent-stage boundaries. No high-impact functionality item requires maintainer discussion.

## Proposed Implementation Shape

### Module Layout And Public Surface

- `src/rphys/datasources/sources.py` owns the framework-neutral sample-source boundary:
  - `SampleRequest`: frozen request record with `requested: tuple[FieldLocator, ...] | None`, `eager: bool`, optional `deterministic_fingerprint`, optional `request_id`, and primitive metadata/invalidation inputs. `requested=None` means all index-linked fields; an empty tuple is invalid to match `SampleBuilder` prevalidation behavior.
  - `SampleRuntimeContext`: frozen primitive evidence for source id/index id, position, entry id, record id, request fingerprint, optional epoch, optional worker id/count, optional rank/world size, optional seed material, and caller metadata. It is evidence, not workflow state.
  - `WorkerContextFactory`: deterministic context factory from base seed, epoch, worker/rank metadata, request fingerprint, and `DataSourceIndexEntry` values. It does not call package-level random APIs or own stochastic augmentation execution.
  - `SampleSource`: public runtime-checkable protocol or small ABC with `__len__`, `sample_at(position, request=None, context=None) -> Sample`, and optional default-request behavior for `__getitem__`. The exact `__getitem__`/`sample_at` split is a public API decision because `SampleRequest` cannot be passed through the normal one-argument Python indexing protocol.
  - `IndexSampleSource`: composes `DataSourceIndex` plus `SampleBuilder`, resolves a position to `IndexItem` and aligned `DataSourceIndexEntry`, derives context, calls `SampleBuilder.build(..., requested=request.requested, eager=request.eager)`, and attaches only inspectable source/context evidence allowed by current `Sample`/`SampleField` contracts.
  - Derived-index behavior initially uses the same implementation path as `IndexSampleSource`. A distinct `DerivedIndexSampleSource` should be added only when it validates derived-index metadata or derived-source provenance from code-backed Stage 8 descriptor assembly; it must not be a name-only subclass, import export operations, or rescan output directories.
- `src/rphys/datasources/cache.py` owns deterministic cache contracts:
  - `CachePolicy`, `CacheKey`, `CacheContext`, `CacheEntry`, `CacheManifest`, `CacheLookupResult`, `CacheWriteResult`, `CacheStore`, `LocalCacheStore`, and `CachedSampleSource`.
  - `CacheStore` should be a narrow public protocol for lookup/read/write/commit evidence over manifest-backed entries. `LocalCacheStore` should use temp-write then atomic commit/rename semantics for local paths and reject incomplete, mismatched, stale, or corrupt entries.
  - Cache payload writing should not default to unsafe pickle or hidden object serialization. The proposal is to require an explicit cache value writer/reader strategy or prepared/field-backed record format for cacheable data; the cache key and manifest can be stable before every optimized payload backend exists.
- `src/rphys/datasources/prepared.py` owns prepared and materialization contracts:
  - `PreparedDataManifest`, `PreparedSampleSource`, `PreparedSampleReader`, `OptimizedStorageFormat`, `OptimizedDataPlan`, `MaterializationPlan`, `MaterializationManifest`, `ShardManifest`, `ChunkMetadata`, `AccessPatternPlan`, `RecordLayoutMetadata`, and `BatchCostMetadata`.
  - Prepared manifests are durable, versioned primitive records because prepared data must be reloadable across processes and must prove equivalence to a `SampleRequest`. Backend-specific readers stay behind explicit reader protocols/factories and return FieldLocator-keyed `Sample` objects.
  - Concrete LitData/WebDataset/Zarr/Arrow/Parquet/mmap adapters remain out of scope; backend identity is recorded as metadata, not imported as a core dependency.
- `src/rphys/datasources/torch.py` owns optional framework adapters:
  - `TorchSampleSourceDataset`, `TorchIndexSampleDataset`, `TorchDataLoaderPlan`, and `TorchDataLoaderBuilder`.
  - The module should remain importable without importing `torch` where practical. Torch-specific behavior imports `torch` lazily and raises a typed dependency error when unavailable.
  - Adapters consume `SampleSource`; they do not scan directories, build indexes, export fields, move devices, or format model tuples.
- `src/rphys/data/collation.py` remains the canonical collation implementation. Stage 9 should add `BatchCollater` there or in a tiny adapter module that delegates directly to `collate_samples`; no custom stack/pad/drop/model-specific collation is introduced.
- Public re-exports should stay scoped to owning packages/modules. The root `rphys` package remains empty. `rphys.datasources.__all__` may get only code-backed Stage 9 names after design approval; no placeholder exports are added.

### Data Flow

1. A caller creates or uses a default `SampleRequest`. The request carries requested `FieldLocator`s, eager/lazy behavior, optional deterministic materialization fingerprint, and primitive metadata/invalidation inputs.
2. `IndexSampleSource.sample_at(position, request, context)` validates the position and request, obtains `index[position]` and `index.entry_at(position)`, derives or records `SampleRuntimeContext`, and calls `SampleBuilder.build` without mutating the index or builder.
3. Derived indexes follow the same `IndexSampleSource` path over a `DataSourceIndex` produced by Stage 8 flows until distinct derived-provenance validation is code-backed. Any derived provenance is sidecar/context evidence, not export execution.
4. `CachedSampleSource` computes a `CacheKey` from the request, context, index entry, `FieldView`/resource fingerprints, codec/software/invalidation inputs, and optional deterministic operation fingerprints. It returns a cached compatible sample only when manifest equivalence is proven; otherwise it delegates to the wrapped source and may write through an explicit local cache store policy.
5. `PreparedSampleSource` validates `PreparedDataManifest` equivalence to the request/context and reads through a manifest-backed reader to return the same FieldLocator-keyed `Sample` shape as `IndexSampleSource`.
6. `TorchSampleSourceDataset` delegates item access to `SampleSource`; `TorchDataLoaderBuilder` combines the dataset, explicit loader plan, and `BatchCollater`.
7. `BatchCollater` delegates to existing `collate_samples`, producing `Batch` while preserving FieldLocator keys and LIST-only field policy behavior.
8. Batch planning, loader state, and data-path profile records describe cost/layout/access/cache/materialization evidence for later training and profiling layers without owning trainer or device behavior.

### Dependency Direction

- `rphys.datasources.sources` may import `rphys.datasources.indexes`, `rphys.datasources.index_items`, `rphys.data.sample_builders`, `rphys.data.containers`, `rphys.data.locators`, and primitive IO helpers.
- `rphys.datasources.cache` may import source contracts, `FieldLocator`, `DataSourceIndexEntry`/`IndexItem` descriptors when needed for key construction, `ResourceRef`/`FieldView` descriptors, and primitive freezing helpers. It must not import torch, trainer, metrics, prediction, or concrete optimized-storage SDKs.
- `rphys.datasources.prepared` may import source contracts, `Sample`/`FieldLocator`, datasource/IO descriptors, and operation fingerprint strings as primitive evidence. It must not execute sample/batch operations in `__getitem__` or import trainer/model code.
- `rphys.datasources.torch` imports sample-source contracts but is not imported by `rphys.datasources.sources`, cache, prepared, `SampleBuilder`, or `DataSourceIndex`.
- `rphys.data.collation` stays independent of datasource indexes/cache/prepared/torch. `BatchCollater` can wrap it, but collation remains runtime-field behavior.
- Stage 8 export code may produce derived indexes and field refs consumed by Stage 9, but Stage 9 should not import export operations into core source/cache/prepared modules.

### Extension Points And Internal Helpers

- Public extension points: `SampleSource`, `CacheStore`, explicit cache value reader/writer strategy if approved, `PreparedSampleReader`, optional optimized-storage reader/backend protocol if approved, `BatchCollater`, and importable objects referenced by downstream `_target_` paths.
- Internal helpers: locator/request coercion, request fingerprinting, primitive freezing/thawing, worker/rank context derivation, cache-key payload normalization, local temp-write/commit helpers, cache manifest validation, prepared equivalence checks, shard/chunk/access-plan validators, cost metadata validators, and optional dependency import diagnostics.
- Revisit triggers: public helper/protocol promotion becomes necessary for a second backend, second downstream project, repeated Stage 15 profiling use, cross-process cache/prepared interchange, or stable DDP/streaming coordination.
