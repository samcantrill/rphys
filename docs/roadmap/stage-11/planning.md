## Overview

- What this stage covers: reusable collection/view/collector contracts; `SampleCollection`, `SampleCollectionViewPlan`, `SampleCollectionView`, and sample collector contracts; base `Loss`, `Objective`, and `Metric` contracts; input selector specs; contract metadata for differentiability, reductions, masking, missing-field behavior, gradient expectations, grouping, and observation views; structured result records; metric observation collection records; reusable grouped-view behavior over those records; dependency-light imports and tests.
- Why it exists: later learners/trainers need a typed optimizer scalar and structured terms, while prediction/evaluation/analysis need detached observations and metric observation collections with grouping metadata. This stage prevents losses, objectives, metrics, methods, learners, trainers, sample reconstruction, and evaluators from collapsing into one generic callable.
- Primary outcomes: iterables can be materialized into inspectable collections by collectors; collection views can transform collections into collections; sample collections can group/sort/stitch sample windows before metrics; losses can compute differentiable term records from declared fields; objectives can aggregate loss and optimizer-relevant terms into `ObjectiveResult.total`; metrics can compute detached observations or observation collections; metric observation views can derive coarser or projected observation collections without mutating core metrics; all contracts are field-centric and inspectable.
- Non-goals: concrete numerical algorithms, concrete physiological metrics, automatic tensor backend integration, trainer loop behavior, optimizer stepping, scheduler/checkpoint/device/distributed behavior, datasource scanning/loading, prediction export, evaluation runners, report writing, plotting, logger integrations, and workflow/project configuration.
- Related feature docs: none found.
- Impacted repo areas: `src/rphys/collections.py`, `src/rphys/data/collections.py`, `src/rphys/losses/`, `src/rphys/objectives/`, `src/rphys/metrics/`, possibly `src/rphys/errors.py` for code-backed error bases/specifics, scoped package tests, focused unit tests under `tests/unit/rphys/{data,losses,objectives,metrics}/` and collection tests, contract tests under `tests/contracts/`, synthetic integration tests only if composition is validated, and docs/docstrings.
- Current state: loss/objective/metric package homes exist but export no public names; field roles already exist; broad method/learning/training/evaluation errors exist but no loss/objective/metric-specific broad errors; Stage 10 method/model contracts are planned but unlanded.
- Key uncertainty: resolved through maintainer discussion. Stage 11 adds broad public loss/objective/metric error bases, keeps scalar values as backend-native handles plus metadata, records TorchMetrics/torchmetrics as optional adapter pressure rather than a core import, uses patch-only result `fields`, and represents metric collection entries as row-like `MetricObservation` records.
- User clarification questions and resolved answers: FQ-6 broad public errors approved; DQ-3 raw backend handles approved; TorchMetrics/torchmetrics distributed intent recorded as future optional-adapter/trainer pressure; DQ-4 patch-only `fields` approved; DQ-5 `MetricObservation` collection entries approved; latest refinement replaces the public `MetricResultTable`/aggregation wording with `MetricObservationCollection` and `MetricObservationViewPlan`, promotes reusable `Collection`/`CollectionViewPlan`/`CollectionView` and `Collector`/`CollectorResult` contracts into Stage 11, and includes `SampleCollection`/`SampleCollectionViewPlan`/`SampleCollectionView` before metric phases.
- Planning priority or optimization target: establish a small, dependency-light, field-centric contract layer that Stage 12 and Stage 13 can compose without forcing numerical algorithm catalogs, backend tensor dependencies, trainer behavior, or evaluation/reporting machinery into Stage 11.

## Behavior Confirmation

- Included behavior: Stage 11 defines dependency-light collection/view/collector contracts; `SampleCollection`, `SampleCollectionViewPlan`, `SampleCollectionView`, and sample collector contracts; base contracts and records for `Loss`, `Objective`, and `Metric`; typed `FieldLocator` input specs; loss/objective/metric contracts and contexts; `LossTerm`, `LossResult`, `ObjectiveTerm`, `ObjectiveResult`, `MetricValue`, `MetricObservation`, `MetricObservationCollection`, `GroupBySpec`, a small `MetricObservationViewPlan` descriptor, and explicit view behavior over observations/collections; broad public `RemotePhysLossError`, `RemotePhysObjectiveError`, and `RemotePhysMetricError`; and scoped public exports validated by package tests.
- Default behavior: collectors materialize iterables into collection snapshots and may return `CollectorResult`; collection views transform collections into collections and do not introduce a separate view-result class. Sample collection views group/sort/stitch synthetic sample windows by metadata and emit `SampleCollection` outputs that are iterable over `Sample`. Losses compute differentiable error or penalty terms from declared fields and report differentiability, reduction, masking, missing-field behavior, and gradient expectations descriptively. Objectives aggregate loss and optimizer-relevant terms and expose `ObjectiveResult.total` as the scalar used for backward. Metrics compute detached observations or observation collections, may write declared metric fields, and keep any accumulation or distributed synchronization in explicit view/adapters rather than hidden mutation inside core `Metric`.
- Failure behavior: invalid locators/specs, missing fields, incompatible schemas/types, invalid reductions or masks, invalid totals, invalid collection items, missing group/sort metadata, undeclared or duplicate terms/fields, invalid metric observations/collection entries, ambiguous grouping/views, unsupported backend values, and execution wrapping fail loudly with typed errors and inspectable context. Specific public subclasses are added only when implementation tests prove value.
- Unsupported behavior: concrete loss and metric catalogs, broad public `Measure`, broad input adapters without implementation pressure, optimizer/scheduler/checkpoint/device/distributed/trainer behavior, learner/evaluator/report runners, datasource scanning/loading, sample building from `IndexItem`, concrete signal-processing stitching/resampling/interpolation/filtering algorithms, file writing/export, logger-only state, report/dataframe dependencies, plotting, workflow/project configuration, and hard tensor/backend dependencies in core imports.
- Resume/interruption behavior: planning workflow only; Stage 11 does not define workflow resume, trainer checkpoint, metric log replay, or evaluator restart semantics.
- Downstream implications: Stage 10 prediction fields can feed synthetic losses/metrics when available; Stage 12 can consume `ObjectiveResult.total`, structured terms, and metric observations without string-key guessing; Stage 13 can consume `SampleCollection`/`SampleCollectionView`, metric observation collections/views, and can own evaluation runners, reports, and export decisions.
- Explicit deferrals: exact public class/module layout, concrete specific error subclasses, shared helper extraction, concrete algorithms, backend/tensor helpers, learner/trainer integration, evaluation/reporting/export, first-class metric identity/report fields, and workflow/project configuration are deferred to implementation planning or later roadmap stages.
- Why this behavior is locked: FQ-1 through FQ-9 are resolved by roadmap text, glossary terms, package/import constraints, existing field/container contracts, Stage 10 prerequisite planning evidence, and maintainer approval for FQ-6 on 2026-05-16.

These helpers should not appear in the first phase unless tests prove the result-patch contract needs a public application function.
- Likely internal helpers:
  - Package-local or duplicated private helpers for locator coercion, unique name validation, string-keyed metadata/provenance mapping, primitive descriptor validation, field update validation, context coercion, and execution-error wrapping.
  - Shared private helper extraction is allowed only after identical behavior repeats across at least two packages during implementation. No public shared selector/adapter base should be introduced in Stage 11.
- Data flow:
  - A `Collector` consumes `Iterable[T]` and returns `CollectorResult[C]` where `C` is a collection snapshot. `CollectorResult` carries collection materialization diagnostics such as accepted count, rejected/skipped entries with reasons when an explicit skip/reject policy is configured, metadata, and provenance. The default collector posture is fail-loud on invalid items or missing required metadata; silent dropping is not allowed. It is not an `OperationResult`; operation wrappers may put the collection on `OperationResult.output` and copy diagnostics to metadata/provenance when collection behavior is run in a pipeline.
  - A `CollectionView` consumes a collection and returns a collection. Views do not return a separate result class; view provenance belongs on output collection metadata, output items, or the operation wrapper if one is used.
  - A `SampleCollector` materializes `Iterable[Sample]` into `SampleCollection`. `SampleCollection` exposes iteration over `Sample` values for metric compatibility and an explicit entries/items access pattern, such as an `items` property or equivalent method, that yields `CollectionItem[Sample]` records preserving per-sample metadata/provenance. A `SampleCollectionView` groups by sample item metadata such as `record_id` or `subject_id`, sorts by window/timestamp metadata such as `window_start`, applies declared descriptor-level field stitch/reconstruction behavior with synthetic or injected fake stitch functions only, and emits a `SampleCollection` of reconstructed or selected samples. `SampleCollection` is iterable over `Sample`, so it can feed metric call sites that expect `Iterable[Sample]` while preserving collection provenance through the entries/items API.
  - A `Loss` receives a `Sample`, `Batch`, or `FieldContainer`-compatible object plus optional `LossContext`; it reads declared prediction/target/auxiliary fields, validates schemas/types/masks/reductions, and returns `LossResult`.
  - `LossResult` carries named `LossTerm` records and optional declared field updates under `losses/*`, `diagnostics/*`, or other explicitly declared locators. It does not step optimizers, compute metrics, export files, or mutate inputs by default.
  - An `Objective` receives structured loss results, direct optimizer-relevant auxiliary terms, or declared field/container inputs plus optional `ObjectiveContext`; it returns `ObjectiveResult` with required `.total`, term breakdown, diagnostics, metadata/provenance, and optional declared field updates.
  - A `Metric` receives declared prediction/target/output fields plus optional `MetricContext`; it returns row-like `MetricObservation` records and may include a `MetricObservationCollection`. Metric observations carry `level` or `scope`, grouping metadata, view metadata when applicable, provenance, and detached `MetricValue`s that must not be used as optimizer targets.
  - Observation view behavior receives `MetricObservation` records or `MetricObservationCollection` collections and returns `MetricObservation` records or `MetricObservationCollection` collections with coarser/different `level`/window/grouping metadata and view provenance. Core metrics do not hide stateful accumulation, and observation views do not introduce a second result shape.
- Dependency direction:
  - `rphys.collections` may depend only on `rphys.errors` and standard-library typing/collections.
  - `rphys.data.collections` may depend on public `rphys.collections`, `rphys.data` containers/fields/locators, and central errors.
  - `losses`, `objectives`, and `metrics` may depend on `rphys.data` and `rphys.errors`.
  - `objectives` may depend on public `rphys.losses` result records only when aggregating loss terms is code-backed by tests; losses do not depend on objectives.
  - `metrics` must not depend on `losses` or `objectives` for core behavior.
  - None of `losses`, `objectives`, or `metrics` may import `methods`, `models`, `learning`, `training`, `prediction`, `evaluation`, `analysis`, `datasources`, `io`, `ops.export`, torch, numpy, pandas, scipy, matplotlib, or video stacks on import.
  - Stage 10 artifacts are prerequisite planning evidence, not landed code. Stage 11 should type against existing `FieldContainer`/`Sample`/`Batch` contracts and use synthetic fakes for method-output-shaped data until Stage 10 lands.
- Extension points:
  - Downstream projects extend behavior by implementing structural protocols or importable callable objects referenced by `_target_` paths.
  - Additional concrete loss/metric algorithms can add package-local spec/result extensions later without changing the base protocols.
  - Optional backend adapters can wrap native tensors/scalars outside the core import path later. Stage 11 records backend and differentiability semantics descriptively rather than importing backend types. Maintainer intends TorchMetrics/torchmetrics-backed concrete loss or metric adapters for distributed synchronization when needed; that intent is compatibility pressure for later optional adapters and Stage 12 trainer/distributed behavior, not a Stage 11 core import.
  - Future Stage 13 evaluation/reporting can adapt `SampleCollection` and `MetricObservationCollection` to evaluation runners, dataframes, or reports without making runner/report serialization part of Stage 11.
  - A stateful object that turns an iterator into a collection is a `Collector` or runner-owned buffer, not a `Builder`; `SampleBuilder` already means one lazy `IndexItem` to one runtime `Sample`.
- Compatibility constraints:
  - `ObjectiveResult.total` must remain the only optimizer scalar contract for Stage 12.
  - Metric outputs must remain detached and observation collections must preserve grouping/provenance enough for Stage 13 evaluation/reporting.
  - Public row/schema/import-path choices must let Stage 13 wrap Stage 11 `SampleCollection` reconstruction and post-metric `MetricObservationCollection` views without requiring Stage 11 internals to be edited.
  - Result field updates must be explicit patch data or caller-selected application behavior; no hidden mutation, export, logger state, or durable persistence.
