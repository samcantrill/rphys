## Roadmap Extraction

- Baseline roadmap outcome: define generic operation contracts and pure functional-kernel placement rules before specialized sample, batch, export, metric, or analysis operations are introduced.
- Prerequisites: Stage 1 naming/error vocabulary, Stage 2 runtime `Sample`/`Batch`/`FieldValue`/collation, Stage 3 lazy refs and `IndexItem`, Stage 4 codecs/lazy `SampleField`/`SampleBuilder`, and Stage 5 datasource/index foundations. Current code shows Stages 1-4 are implemented; Stage 5 is represented by an untracked planning artifact and is not yet a code-backed prerequisite in this checkout.
- Prior or adjacent roadmap links: Stage 5 supplies datasource/index behavior that should stay outside operations; Stage 7 will add `SampleOp`, `BatchOp`, transforms, augmentations, checks, and pipelines over this foundation; Stage 8 will model export/save flows as operations; Stage 9+ will use operation/batch behavior in training-facing loaders and cache paths.
- Primary feature docs: none found; `docs/roadmap.md` is canonical.
- Deferred or out-of-scope roadmap work: concrete CHROM/POS/model-specific preprocessing kernels, `SampleOp`/`BatchOp` contracts, transforms, augmentations, checks, export/save ops, derived datasource building, Torch adapters/loaders/cache, models/methods/losses/objectives/metrics/learners/trainers, real datasource preprocessing pipelines, workflow/artifact runtimes, broad `BatchProgram` or universal batch execution language, hidden randomness/device movement/schema conversion, and heavy optional numerical dependencies unless explicitly approved later.
- Compatibility obligations: preserve lightweight imports; keep functional kernels callable without runtime containers; prevent kernels from depending on `Sample`, `Batch`, `FieldRef`, `IndexItem`, datasource objects, file IO, hidden randomness, hidden device movement, or hidden schema conversion; keep operation classes responsible for contracts, context, provenance, declared mutation, side effects, and pipeline composition; make input/output expectations, mutation, side effects, context requirements, and failure modes inspectable; defer deterministic/randomness declaration fields until later operation families need them; fail loudly through typed operation or pipeline errors when contracts are invalid.
- Public surfaces or durable artifacts likely affected: likely focused `rphys.ops` submodules or package exports for `Operation`, `OperationContract`, `OperationContext`, `OperationResult`, `OperationPipeline`, `OperationRole`, and `FunctionalKernel`; likely exercised concrete operation/pipeline errors in `src/rphys/errors.py`; likely package import-boundary tests, source-mirrored unit tests under `tests/unit/rphys/ops/`, public contract tests under `tests/contracts/`, and possible synthetic integration tests once operations compose with runtime containers. No durable file artifact is directly required by Milestone 6.

## Overview

- What this stage covers: generic context-aware operation contracts, operation metadata/results, operation roles, operation pipeline composition, contract validation, functional-kernel wrapper boundaries, context requirements, declared mutation and side-effect metadata, and failure-mode reporting.
- Why it exists: later sample transforms, batch transforms, export ops, metrics, analysis, and training-facing flows need a strict reusable operation layer that separates pure numerical kernels from runtime-container, datasource, IO, randomness, device, provenance, and pipeline concerns.
- Primary outcomes: functional kernels remain plain callables over payloads, parameters, and explicit metadata; operation objects can declare input/output types, mutation policy, side effects, context requirements, and failure modes; deterministic/randomness declaration fields are deferred; operation pipelines propagate context and validate contracts; no algorithm-specific preprocessing kernels are required.
- Non-goals: concrete rPPG algorithms, CHROM/POS preprocessing, `SampleOp`/`BatchOp` APIs, transform/augmentation/check catalogs, export/save/derived datasource flows, datasource/index building, codec loading/saving, data loader/cache/training adapters, model formatting, workflow stages/artifacts, hidden global registries, hidden randomness/device movement/schema conversion, and universal batch-program languages.
- Related feature docs: none found.
- Impacted repo areas: `src/rphys/ops/`; `src/rphys/errors.py`; `tests/package/`; likely `tests/unit/rphys/ops/`; likely `tests/contracts/`; possibly `tests/integration/` for small dependency-light pipeline composition examples; docs/docstrings/glossary only if public operation wording needs alignment.
- Current state: `rphys.ops` is an empty public package home with no operation names exported. Broad operation and pipeline error bases exist. Runtime containers and lazy sample fields are implemented and tested. Codec and sample-builder contracts are implemented and datasource-neutral. No operation-specific tests or implementation exist.
- Key uncertainty: implementation planning may still refine type-variable spelling, concrete exercised error names, private helper factoring, and exact validation command grouping, but only within the locked functionality and design boundaries.
- User clarification questions and resolved answers: no open maintainer clarification remains for plan quality. Maintainer approvals are recorded for wrapper-first/package re-exports, minimal contract fields with deterministic/randomness declarations excluded, runtime-only context/result records, result-return execution, and sequence-only pipeline construction.
- Planning priority or optimization target: establish the smallest dependency-light operation/kernel foundation that later Stage 7+ runtime operations can build on without weakening scientific provenance, mutation visibility, import boundaries, or separation from datasource/IO/workflow concerns.

## Behavior Confirmation

- Included behavior: Stage 6 defines a generic `rphys.ops` operation foundation: code-backed operation public surface, pure functional-kernel boundary, inspectable operation contracts, explicit `OperationContext`/`OperationResult` behavior, generic execution with declared mutation and side effects, ordered `OperationPipeline` composition, typed operation/pipeline failures, cross-stage boundary preservation, and focused validation/docs.
- Default behavior: Operations are explicit, dependency-light, and provisional. Functional kernels are plain callables over payloads, parameters, and explicit metadata. Operation wrappers are used when contract declarations, context, provenance/result metadata, mutation policy, side-effect declarations, or pipeline composition are needed. Pipelines are ordered/sequential by default and validate contracts while propagating explicit context.
- Failure behavior: Invalid contracts, invalid context, invalid results, contract mismatch, unsupported mutation/side effects, and pipeline incompatibility fail loudly through broad `RemotePhysOperationError`/`RemotePhysPipelineError` bases plus only concrete errors exercised by public behavior and tests. Failures should carry structured context such as operation, step, expected, actual, role, and cause where useful.
- Unsupported behavior: Stage 6 does not implement SampleOp/BatchOp locator permissions, transforms, augmentations, checks, export/save ops, datasource scanning/index/manifests, codec IO, cache/materialization keys, trainer/data-loader behavior, workflow/artifact runtimes, concrete rPPG kernels, hidden global registries, hidden randomness, hidden device movement, hidden schema conversion, or broad batch-program/DAG execution languages.
- Resume/interruption behavior: Stage 6 adds no workflow runtime or resume semantics. Planning resumes from this artifact; implementation later should keep operation execution ordered and inspectable but does not own run resume/retry orchestration.
- Downstream implications: Stage 7 can layer SampleOp/BatchOp-specific field-locator, RNG, routing, and equivalence behavior on top of the generic operation foundation. Stage 8 can model export/save operations over the same foundation without making Stage 6 own export targets or derived datasource manifests. Stage 9+ can add cache/materialization identity after Stage 5/7/8 contracts are settled.
- Explicit deferrals: Exact public API/module split, dataclass/protocol fields, type-variable spelling, kernel-wrapper ergonomics, pipeline validation mechanics, and concrete exercised error names are design-gate work. Stage 5 index identity/manifest/composite decisions remain adjacent and are not settled here. Stage 7/8 context/result schemas remain deferred to their milestones.
- Why this behavior is locked: FQ-1 through FQ-8 are dependency ordered and repo-resolved by the Milestone 6 roadmap, glossary vocabulary, current Stage 2/4 code contracts, Stage 5 adjacent uncertainty, and future Milestone 7/8 boundaries. No high-impact functionality ambiguity or maintainer decision remains before design proposal.

## Proposed Implementation Shape

- Likely modules or packages: keep Stage 6 under `src/rphys/ops/` with narrow implementation-home submodules such as `contracts.py` for declarations, `context.py` for generic context/result records, `kernels.py` for the plain-callable `FunctionalKernel` boundary, `core.py` for the executable operation wrapper/contract validation path, `pipelines.py` for ordered generic composition, and private `_validation.py`/`_typing.py` helpers if needed. Re-export only code-backed provisional names from `rphys.ops.__init__` and make package-level `rphys.ops` imports the preferred public path in docs/tests; do not add root `rphys` convenience exports, `rphys.transforms`, workflow packages, registries, or placeholder subpackages for Stage 7/8 features.
- Likely public classes/functions/protocols: `Operation`, `OperationContract`, `OperationContext`, `OperationResult`, `OperationPipeline`, `OperationRole`, and `FunctionalKernel` as the roadmap-named provisional surface. Likely small supporting public declarations may include a mutation-policy value if the implementation cannot express mutation clearly with a boolean/string field, but adding extra public names should be justified by tests. Prefer composition: an `Operation` wrapper owns a callable plus contract rather than requiring subclass inheritance. Custom downstream behavior can be provided as importable callables or operation objects, not registry entries. Public Protocol/base introduction is explicitly deferred until repeated custom operation-object needs appear.
- Likely internal helpers: private coercion and validation helpers for operation names/roles, runtime type expectations, context requirement keys, mapping immutability, result validation, side-effect labels, adjacent contract compatibility, and step-aware error context. Keep helper imports within `rphys.ops` and `rphys.errors`; do not import `rphys.io`, `rphys.datasources`, `tests.support`, or heavy optional modules from operation core modules.
- Data flow: users call a plain kernel directly when no contracts/context/provenance/pipeline behavior is needed. When operation behavior is needed, an `Operation` receives explicit input and optional `OperationContext`, validates input/context against `OperationContract`, invokes its callable with explicit values, wraps the output and metadata/provenance in `OperationResult`, validates the result/output contract, and returns `OperationResult` from both `.run()` and `__call__` or raises a typed operation error. Stage 6 does not add a separate raw-output execution API. `OperationPipeline` stores an ordered tuple of operations, validates adjacent output/input compatibility where declarations are present, propagates the same explicit context unless the caller provides per-step overrides later, feeds each step `result.output` to the next step, and raises typed pipeline errors with step index/name and cause.
- Dependency direction: `rphys.ops` may depend on Python stdlib typing/dataclasses/enums/collections and `rphys.errors`. It should treat `Sample`, `Batch`, and `SampleField` as ordinary user payload types when callers pass them, not import them in core contracts. Stage 7+ sample/batch modules may import Stage 6 operation contracts; Stage 6 must not import Stage 7 modules, datasource/index/manifest code, codec IO, export/save modules, training loaders, cache/materialization, or downstream workflow runtimes.
- Extension points: direct importable kernel callables; `Operation` instances wrapping callables with contracts; structural operation-like objects only if they expose the same minimal `contract` and call/result behavior without inheritance; pipeline construction from ordered operation sequences. Future SampleOps/BatchOps/export ops should extend by composing or specializing these contracts in later submodules, not by editing private Stage 6 helpers or relying on hidden registries.
- Compatibility constraints: all public names are provisional and code-backed; functional kernels stay container-free; operation contexts/results stay generic and do not settle Stage 5 identity, Stage 7 sample/batch context, Stage 8 export result, or Stage 9 cache/materialization schemas; no hidden RNG/device movement/schema conversion; no field-locator read/write/delete permissions; no broad batch-program/DAG language; imports remain lightweight and package tests must prove no optional video/array/plotting/deep-learning/dataset stacks load.
