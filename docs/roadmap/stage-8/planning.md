## Roadmap Extraction

- Baseline roadmap outcome: represent formatting, symlinking, derived fields, and prediction export as normal operation pipeline flows that write fields through codecs and emit derived `DataSourceRef`s.
- Prerequisites: Stage 4 codec save/load/probe contracts and lazy sample bridge; Stage 5 datasource/index descriptors and manifests; Stage 6 generic operation contracts and result semantics; Stage 7 code-backed `OperationStep` foundation plus later sample/batch operation specialization. Stages 4-7 Phase 1 are code-backed in this checkout; Stage 8 plans against the landed `OperationStep` interface and keeps sample/batch specialization as a separate compatibility check, not a blocker.
- Prior or adjacent roadmap links: Stage 4 provides `SaveContext(target: FieldRef)` and `CodecSaveResult`; Stage 5 provides descriptor/index/manifest provenance but defers export; Stage 6 provides generic operation wrappers but defers export-specific result schemas; Stage 7 provides the `OperationStep` composition interface and generic pipeline result forwarding that Stage 8 export operations must satisfy; Stage 9 will consume prepared/indexed data without owning export; Stage 13 will export durable predictions through Stage 8 mechanisms.
- Primary feature docs: none found; `docs/roadmap.md` is canonical.
- Deferred or out-of-scope roadmap work: Torch adapters/data loaders/cache, prepared training-data manifests, prediction/evaluation/analysis/reporting, Method/Learner/Trainer artifact writing, model formatting, metric computation, project workflow stages/artifact stores, real datasource preprocessing pipelines, concrete codec catalog, broad cache invalidation, and generic workflow resume/retry/runtime behavior.
- Compatibility obligations: preserve lightweight imports; keep datasource adapters from invoking export code; keep `FieldCodec.save` as single-field codec IO; avoid rescanning output directories to build derived refs; preserve source `ResourceRef` lineage for symlink/copy policy; make output layout deterministic; make idempotency policy explicit and tested; count written/skipped/linked/copied/replaced/failed fields; keep Method/Learner/Trainer/Loss/Objective/Metric from implicitly writing durable artifacts.
- Public surfaces or durable artifacts likely affected: likely `rphys.ops.export` or equivalent export operation home; likely `rphys.io` reuse of codec save contexts/results; likely `rphys.datasources` additions for derived datasource construction; likely public code-facing full `*Operation` names rather than `*Op` shorthand; descriptor outputs that include derived `FieldRef`, `RecordRef`, and `DataSourceRef` evidence. DQ-5 and DQ-6 defer new durable report/derived-manifest file schemas out of Stage 8.

## Overview

- What this stage covers: export specs/targets/layout policy, codec-selection annotation, save/export operation behavior, field/record/export result records, export reporting, deterministic target layout, idempotency policy, derived descriptor assembly, derived `DataSourceRef` construction, and round-trip reload through normal datasource/index/sample machinery.
- Why it exists: deterministic preprocessing, formatting, symlinking/copying, materialized derived fields, and durable predictions need to be represented as explicit library operations rather than hidden datasource adapter, trainer, or method side effects.
- Primary outcomes: a datasource/index can be processed by an operation pipeline and exported into a derived `DataSourceRef`; export results report per-field and aggregate counts; layout and idempotency are deterministic; derived refs can be filtered, indexed, and loaded again through existing machinery.
- Non-goals: datasource scanning, split/group/index construction beyond consuming existing descriptors, codec implementation catalog, generic artifact stores, workflow stages, trainer/prediction runner behavior, metric/report generation, Torch data loading/cache/materialization, model-format adapters, and ad hoc output-directory rescans.
- Related feature docs: none found.
- Impacted repo areas: likely `src/rphys/ops/` for export operation contracts; `src/rphys/io/codecs.py` only by composition unless design finds a narrow additive need; `src/rphys/datasources/` for derived datasource manifest/ref assembly; `src/rphys/errors.py`; `tests/package/`; `tests/unit/rphys/ops/`; `tests/unit/rphys/datasources/`; `tests/contracts/`; `tests/integration/`; `tests/support/` synthetic codecs/datasources.
- Current state: codec-level save works through `CodecRegistry.save(FieldValue, SaveContext(target: FieldRef))` and returns `CodecSaveResult`; datasource refs and index manifests exist but no derived datasource writer exists; generic `OperationStep`, `Operation`, and `OperationPipeline` are code-backed, but no export-specific operation family exists.
- Key uncertainty: pending implementation mechanics include concrete exercised error names, manager-synthesized partial-failure policy details, and whether export later adds optional adapters for Stage 7 sample/batch specialization. Public import/name shape is locked by DQ-1, operation composition is locked by DQ-2 against the landed `OperationStep` contract, target layout/fingerprint/idempotency semantics are locked by DQ-3, report durability is locked in-memory-only by DQ-5, derived manifest scope is locked descriptor-only by DQ-6, link/copy lineage/storage scope is locked by DQ-7, and the old Stage 6 subclass-versus-factory operation-shape uncertainty is resolved.
- User clarification questions and resolved answers: maintainer clarified during context scaffold that Stage 8 should prefer full public code-facing names using `Operation` rather than abbreviated `Op`; roadmap names such as `SaveOp` and `CodecSelectionOp` are shorthand unless later design evidence requires otherwise. No additional maintainer questions were asked in this pass.
- Planning priority or optimization target: prove the smallest dependency-light synthetic export round trip from an existing datasource/index through operation-driven save into a derived `DataSourceRef` that can be indexed and loaded again, while preserving provenance, idempotency, layout determinism, and import boundaries.

## Behavior Confirmation

- Included behavior: Stage 8 defines explicit export request data, deterministic output layout, export/idempotency policies, side-effect-free codec-selection annotation, `SaveOperation` writing through field codecs, field/record/export result records and reports, derived datasource assembly without rescanning, symlink/copy lineage preservation, and a synthetic export-to-reload proof.
- Default behavior: export is explicit and operation-driven; specs are data; codec selection does not write; save writes declared fields only through existing codec save contracts; existing targets are not silently overwritten; derived descriptors are assembled from export results; public code-facing operation names use full `Operation` wording.
- Failure behavior: missing fields, invalid targets/layouts, unsupported or ambiguous codecs, idempotency conflicts, failed codec saves, missing source lineage for link/copy, partial record failures, manifest assembly errors, and boundary violations fail loudly with typed errors or typed failed-result evidence.
- Unsupported behavior: datasource adapters invoking export, output-directory rescans to infer derived refs, implicit artifact writes by Method/Learner/Trainer/Loss/Objective/Metric, prediction/evaluation/metric computation, cache/materialization manifests, Torch loaders, workflow stages/artifacts, generic retry/resume runtime, concrete codec catalogs, heavy optional imports, and raw datasets.
- Resume/interruption behavior: planning workflow only; Stage 8 idempotency is export behavior, not a generic workflow resume engine.
- Downstream implications: Stage 8 enables Stage 13 durable predictions and Stage 14 synthetic smoke reloads by making exported fields normal datasource refs.
- Explicit deferrals: exact API shape, module layout, operation inheritance/wrapping, durable report serialization, durable derived manifest serialization, target URI template details, concrete error names, and validation coverage are deferred to later planning gates. DQ-5 resolves the Stage 8 report side as in-memory-only, DQ-6 resolves the Stage 8 derived datasource side as descriptor assembly only, and DQ-7 resolves link/copy lineage and storage-adapter scope.
- Why this behavior is locked: FQ-0 is a direct maintainer clarification and FQ-1 through FQ-8 are repo-resolved from roadmap, glossary, adjacent-stage contracts, and current code evidence. No high-impact functionality queue item needs maintainer discussion.

## Proposed Implementation Shape

- Likely modules or packages:
  - `rphys.ops.export` as the export operation home for request records, layout/policy records, `CodecSelectionOperation`, `SaveOperation`, field/record/export result records, and export reports.
  - `rphys.datasources.derived` as the descriptor assembly home for `DerivedDataSourceBuilder`, a narrow `DerivedDataSourceAssembly` product, and any roadmap-named `DataSourceManifestWriter` adapter only if it remains descriptor-only. Stage 8 does not introduce a public derived datasource manifest file schema.
  - `rphys.io.codecs` remains the codec save contract home. Stage 8 composes `CodecRegistry.save(value, SaveContext(target=...))` and `CodecSaveResult`; it does not move export layout, record provenance, idempotency, or manifest writing into codec contexts.
  - `rphys.errors` receives only exercised Stage 8 typed errors if needed, such as invalid export spec/layout/policy, export target conflict, export execution, and derived datasource assembly failures. Exact names remain design-agreement items.
- Likely public classes/functions/protocols:
  - `ExportSpec`: immutable data describing export id, fields to export, target policy/layout, codec selection requests, metadata-save policy, and optional per-field overrides. It is not an orchestrator and does not hold a codec registry, datasource adapter, runner, cache, or workflow state.
  - `ExportTarget`: immutable target root or resource descriptor plus target namespace metadata. It describes where targets may be derived, not how to scan outputs.
  - `OutputLayout`: deterministic target-derivation object that uses datasource id, record id, field key, export id, and an export spec fingerprint to produce one logical target `FieldRef` per field.
  - `ExportPolicy` and `IdempotencyPolicy`: explicit behavior for existing targets, link/copy/encode requests, explicit copy fallback, partial failures, and count categories. Default policy should avoid silent overwrite and hidden link-to-copy fallback.
  - `CodecSelectionOperation`: side-effect-free public `OperationStep` implementation that validates declared fields, target derivation, codec save support, and metadata-save policy, then returns an inspectable selection/plan result or annotations without writing files.
  - `SaveOperation`: public `OperationStep` implementation that consumes selected/planned export inputs, reads declared runtime fields, creates one `SaveContext(target=FieldRef)` per logical field, invokes `CodecRegistry.save`, applies idempotency/link/copy policy, and returns `OperationResult` whose `.output` is the export result record.
  - `FieldExportResult`, `RecordExportResult`, `ExportResult`, and `ExportReport`: typed in-memory result records that expose outcome status and counts for written, skipped, linked, copied, replaced, and failed fields; carry `CodecSaveResult` evidence where a codec write happened; record ordered source and target `ResourceRef` lineage for link/copy outcomes; and preserve failures as typed evidence rather than log-only text.
  - `DerivedDataSourceAssembly`: typed in-memory descriptor assembly product containing a derived `DataSourceRef`, ordered derived `RecordRef`s, source/export evidence needed for validation, and no new public manifest file schema.
  - `DerivedDataSourceBuilder`: higher-level assembly adapter that consumes successful export results and returns a `DerivedDataSourceAssembly` usable by existing datasource/index/sample machinery.
  - `DataSourceManifestWriter`, if retained from roadmap naming, is a descriptor assembler over existing ref serialization in Stage 8, not a durable writer for a new public manifest format.
  - No public protocol/base class should be added unless design agreement finds a concrete repeated adapter need. Storage, codec, and operation extension should primarily use importable Python objects and existing structural codec support.
- Likely internal helpers:
  - Private validators for export ids, requested fields, target roots/resources, policy combinations, count categories, and failure records.
  - Private fingerprint helper over stable primitive `ExportSpec` content, following Stage 5 manifest checksum style without importing Stage 5 index-manifest policy into export.
  - Private target derivation helper that produces a candidate `FieldRef` and `SaveContext` for one declared field.
  - Private result aggregation helper that folds field outcomes into record/export/report counts.
  - Private local filesystem link/copy helpers and lineage helpers that copy source `ResourceRef` evidence into link/copy results, reject missing source lineage when policy requires it, reject unsupported protocols/cross-protocol link/copy, and allow symlink-to-copy fallback only when policy explicitly requests it.
- Data flow:
  1. A datasource/index/sample flow produces runtime field values plus `RecordRef`/field identity evidence through existing machinery.
  2. `CodecSelectionOperation` consumes `ExportSpec`, the record/sample export input, `CodecRegistry`, and target/layout policy, then resolves target `FieldRef`s and codec support without writing.
  3. `SaveOperation` consumes the selected targets and field values, applies idempotency policy, and saves each field through `CodecRegistry.save(value, SaveContext(target=target, metadata_policy=...))`.
  4. `SaveOperation` returns an `OperationResult`; callers inspect `result.output` for `RecordExportResult` or `ExportResult` and may inspect side-effect evidence only as secondary Stage 6 evidence.
  5. `DerivedDataSourceBuilder` consumes successful field results and assembles derived `RecordRef`s, a derived `DataSourceRef`, and a `DerivedDataSourceAssembly` directly from result evidence, not from filesystem scans or a new manifest file schema.
  6. Existing Stage 5 candidate/index/sample-builder paths consume the derived refs for reload tests.
- Dependency direction:
  - `rphys.ops.export` may import `rphys.io` descriptors/codecs and `rphys.datasources.refs` descriptor types because export bridges runtime operations to descriptors.
  - `rphys.datasources.derived` may import descriptor types and Stage 8 result records, but datasource adapters, filters, splits, indexes, and validation must not import export operations. It must not depend on or emit `DataSourceIndexManifest` as a derived datasource manifest.
  - `rphys.io.codecs` must not import `rphys.ops.export` or `rphys.datasources.derived`; `SaveContext(target: FieldRef)` and `CodecSaveResult` remain single-field IO contracts.
  - `rphys.ops` generic Stage 6 code should not import `rphys.ops.export`; export is a downstream specialization.
  - Methods, learners, trainers, losses, objectives, metrics, prediction, evaluation, and analysis may later call or consume export surfaces explicitly, but must not gain implicit artifact-writing side effects in Stage 8.
- Extension points:
  - Downstream code can provide codecs through explicit `CodecRegistry` instances and structural codec support predicates.
  - Downstream code can choose layout/policy objects and importable copy/link/save adapters where needed; avoid global registries and symbolic config names in Stage 8.
  - Future Stage 13 prediction export should reuse the same field-export path by passing prediction fields as normal runtime fields.
  - Future Stage 9 cache/loading and Stage 15 materialization manifests should consume derived refs/reports additively; Stage 8 should not predefine cache keys, shard manifests, dataloader state, or workflow retry/resume policy.
- Stage 7 compatibility stance:
  - Stage 8 plans against the landed Stage 7 operation-foundation code: public runtime-checkable `OperationStep` with `name`, `contract`, `run(input_value, context) -> OperationResult`, `Operation` implementing `OperationStep`, and generic `OperationPipeline` accepting ordered `OperationStep` sequences while preserving Stage 6 result-forwarding and mapping-rejection behavior.
  - `CodecSelectionOperation` and `SaveOperation` should be real public `OperationStep` objects, not public factories whose main job is to hide a generic `Operation` instance. They may share private helpers or internal configured `Operation` machinery if implementation benefits, but public composition is through the operation-step interface.
  - Stage 8 should not reimplement Stage 7 runtime-operation semantics. Since Stage 7 sample/batch surfaces are present, recheck only whether export steps should additionally adapt to those specialized contracts; do not block the export surface on sample/batch inheritance.
- Compatibility constraints:
  - Preserve existing Stage 4 `SaveContext(target: FieldRef)`, `MetadataSavePolicy`, `CodecSaveResult`, and single-logical-field codec save semantics.
  - Preserve Stage 5 descriptor/index purity: indexes and manifests describe existing refs and item sidecars; derived datasource assembly must not mutate source descriptors or infer records by rescanning outputs.
  - Preserve Stage 6/7 result-return behavior: operation-step execution returns `OperationResult`, pipelines feed `.output`, and Stage 8 must not introduce a raw-output execution bypass.
  - Keep public imports code-backed, narrow, and dependency-light; no root `rphys` exports unless later design agreement explicitly approves them.
