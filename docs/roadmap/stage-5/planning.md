## Overview

- What this stage covers: datasource specs and adapters, scan outputs, validation reports, datasource view plans/results, filters/filter chains, grouping and split builders/results, index plans/builders/results, data source indexes, index build reports, index codecs/manifests, and source-aware composite index behavior around lazy fields.
- Why it exists: later synthetic and real vertical slices need discoverable, validated, filtered, split-safe, indexable datasource references before operation pipelines, export, Torch loading, cache, and training can compose over them.
- Primary outcomes: synthetic `DataSourceAdapter` emits multiple logical fields; view/filtering avoids descriptor mutation; grouping and split assignment produce counts, warnings, and leakage checks; index construction emits lazy `IndexItem`s with datasource/record/group/split/window provenance; durable indexes round-trip through schema-versioned JSON/JSONL-like manifests; `DataSourceIndex` exposes length, item access, iteration, fingerprint, and metadata; source-aware composite indexes preserve child index identity and local item positions.

- Related feature docs: none found.
- Impacted repo areas: `src/rphys/datasources/`; likely new `src/rphys/datasources/adapters.py`, `filters.py`, `splits.py`, `validation.py`, `indexes.py`, or package submodules; `src/rphys/errors.py`; `tests/package/`; `tests/contracts/`; `tests/unit/rphys/datasources/`; `tests/support/` for synthetic datasource/index fixtures; possibly `tests/integration/` for a synthetic scan-to-index-to-sample-builder path; docs/docstrings/glossary only if public wording needs alignment.

## Behavior Confirmation

- Included behavior: Stage 5 includes datasource specs and adapters that scan declared discovery inputs into descriptor refs and scan/validation evidence; typed validation reports with explicit no-IO, probe-only, or explicit-validation-IO policy; non-mutating datasource views, filters, and filter chains over records, fields, views, and index candidates; candidate construction and candidate selection before grouping/splitting; group plans/results and group-aware split plans/results over selected candidates with counts, warnings, rejections, and leakage checks; index plans/builders/results/reports that emit ordered lazy `IndexItem`s with `FieldView`s only; `DataSourceIndex` length, item access, iteration, fingerprint, and metadata behavior; schema-versioned JSON/JSONL-like index manifests and index codecs; source-aware composite indexes preserving child/source identity and local item position; exercised typed diagnostics, lightweight imports, and a synthetic proof path.
- Default behavior: Stage 5 composes new datasource/index objects around Stage 3/4 descriptors instead of adding scan/filter/build/manifest/fingerprint fields or methods to `DataSourceRef`, `RecordRef`, `DataSourceSchema`, `FieldRef`, `FieldView`, `ResourceRef`, or `IndexItem`. Discovery and indexing use explicit importable objects by default, not hidden global discovery. Filters return new views/results and report rejected records/items/candidates. Candidate filters run before split assignment and must be split-agnostic; group and split builders then operate on accepted candidates using datasource/record/candidate provenance. Group-disjoint split behavior is explicit in split plans/results. Index items stay payload-free, transform-free, role-qualified, and compatible with the existing one-item `SampleBuilder`. Durable index manifests own schema-version envelope, stable datasource-index item identity, resource/index canonicalization, and fingerprint policy around descriptor dicts.
- Failure behavior: Missing or invalid datasource inputs, fields, resources, schemas, metadata, group keys, split plans, index requests, manifests, schema versions, fingerprints, item identities, unsupported IO policies, hidden full-load fallback attempts, and leakage violations fail loudly or are reported as typed warnings/rejections according to explicit plan policy. Unsupported or ambiguous manifests/codecs are rejected rather than best-effort loaded. Payload decoding is not performed unless an explicit probe or validation hook allows it.
- Unsupported behavior: Stage 5 does not implement operations, functional kernels, SampleOps, BatchOps, transforms, augmentations, export/save ops, derived datasource writing, Torch `Dataset`/DataLoader adapters, caches, prepared-data manifests, training loops, prediction/evaluation/analysis/reporting, real datasource-heavy SDK integrations, full preprocessing pipelines, concrete rPPG algorithms/models/losses/metrics, workflow/artifact/stage runtimes, project configuration, public fake dataset packages, root convenience exports, or mutation of existing descriptor semantics.
- Resume/interruption behavior: This planning workflow remains docs-only. Runtime resume/retry semantics for datasource scans, validation, manifest writes, or index builds are not introduced as generic workflow behavior in Stage 5. Persisted index manifests provide durable index reconstruction; broader workflow resume, artifact storage, scheduling, retries, and project orchestration remain downstream or `loom` concerns.
- Downstream implications: Stage 6/7 operations can consume lazy samples produced from Stage 5 indexes without redefining datasource or candidate selection. Stage 8 export can wrap codec save and later derived datasource behavior without inheriting Stage 5 export policy. Stage 9 Torch/cache work can wrap `DataSourceIndex` and manifest fingerprints as inputs without treating descriptor equality as cache identity. Real datasource adapters can reuse the spec/adapter/validation/view/candidate/split/index contracts after the synthetic proof path works.
- Explicit deferrals: Plan/result field names beyond locked scan/filter/report/manifest basics, adapter alias/registry policy, concrete error names, and stability labels remain design-gate decisions unless they would alter the locked behavior. Stage 5 import layout is locked submodule-first by DQ-1, scan result shape by DQ-2, validation report shape by DQ-3, typed selection/filter-candidate scope by DQ-4, candidate-level group/split policy by DQ-5, sidecar index-entry identity plus field-native window scope by DQ-6, and manifest envelope/fingerprint policy by DQ-7. Real datasource adapters, rich payload validation, export manifests, cache manifests, Torch loading, and workflow orchestration are deferred to later roadmap stages.
- Why this behavior is locked: The roadmap, glossary, Stage 3 descriptor deferrals, Stage 4 `SampleBuilder` boundary, current datasource docstrings/tests, and immediate implementation order all point to the same functionality: create a dependency-light datasource layer that discovers, validates, filters, groups, splits, indexes, persists, and composes lazy logical fields while preserving descriptor purity and fail-loud scientific provenance. No functionality-level maintainer ambiguity remains.

## Proposed Implementation Shape

- Likely modules or packages:
  - Keep existing descriptor modules unchanged: `rphys.datasources.refs`, `rphys.datasources.schemas`, and `rphys.datasources.index_items`.
  - Add focused Stage 5 modules that match roadmap package homes: `rphys.datasources.adapters`, `rphys.datasources.validation`, `rphys.datasources.filters`, `rphys.datasources.splits`, and `rphys.datasources.indexes`.
  - Keep manifest/index-codec behavior inside `rphys.datasources.indexes` initially, or a private implementation submodule under it, unless the implementation becomes too large to review. Do not add `rphys.datasets`, root `rphys` exports, workflow packages, cache packages, or export/materialization packages in Stage 5.
- Likely public and provisional classes/functions/protocols:
  - `rphys.datasources.adapters`: `DataSourceSpec`, structural `DataSourceAdapter`, and a provisional scan result record such as `DataSourceScanResult`.
  - `rphys.datasources.validation`: `DataSourceValidationReport`, `ValidationIssue`, and a small `ValidationIOPolicy` enum with `no_io`, `probe_only`, and `explicit_validation_io`.
  - `rphys.datasources.filters`: `DataSourceViewPlan`, `DataSourceView`, `DataSourceViewBuilder`, `DataSourceViewResult`, structural `Filter`, `FilterChain`, `FilterResult`, shared selection context/reporting helpers, and pre-index typed target records only when concrete filters need them.
  - `rphys.datasources.splits`: `GroupPlan`, `GroupBuilder`, `GroupResult`, `SplitPlan`, `SplitBuilder`, and `SplitResult`, with multiple named group assignments and group-disjoint behavior over explicit selected split-group keys rather than inferred from split names.
  - `rphys.datasources.indexes`: `IndexPlan`, `IndexCandidate`, a candidate selection view/result if needed, `IndexBuilder`, `IndexResult`, `IndexBuildReport`, `DataSourceIndex`, a minimal index-entry/identity record if needed to keep `IndexItem` pure, `DataSourceIndexManifest`, `DataSourceIndexCodec`, and source-aware `CompositeDataSourceIndex`. Defer a separate public `ConcatDataSourceIndex` unless implementation or later roadmap evidence proves it is distinct.
  - Parent package exports remain a design question. Recommended default is submodule-first public imports for new Stage 5 names while preserving current `rphys.datasources` descriptor re-exports until the manager explicitly approves broader package re-exports.
- Likely internal helpers:
  - Private `_coerce_*`, `_freeze_*`, `_require_*`, `_copy_*`, `_canonical_manifest_json`, and `_fingerprint_*` helpers local to their owning modules.
  - Private report builders for counts, warnings, rejected IDs, and provenance joins. Do not document helper names or make them hidden extension points.
- Data flow:
  - `DataSourceSpec` describes discovery inputs, expected schema/source metadata, include/exclude rules, and declared validation policy.
  - An explicit adapter object scans a spec into a scan result: `DataSourceRef`, `RecordRef`s, descriptor-only field refs, source/scan metadata, and validation evidence.
  - A validation report records missing/invalid datasource, record, field, resource, schema, metadata, and optional probe evidence without hidden payload loads.
  - A reusable typed selection pipeline narrows each stage without mutating descriptors: datasource/view/record/field selection happens before candidate construction; candidate selection happens after window/subset construction and before group/split assignment; group/split assignment happens over selected candidates before final entry creation.
  - A view builder and filter chain derive non-mutating `DataSourceView`s and filter reports from scan/view inputs, excluding invalid datasource/record/field units based on descriptor attributes, metadata, validation evidence, or explicit probe evidence.
  - `IndexBuilder` or an index candidate builder consumes a selected datasource view and an `IndexPlan`, then expands valid units into `IndexCandidate`s representing prospective windows/subsets and locator-to-`FieldView` mappings.
  - Candidate filters run over candidate attributes, source provenance, primitive metadata, and validation/probe evidence. The selected candidate view/result is the input to grouping and splitting.
  - Group builders assign one or more named groups per selected candidate from metadata/provenance. Split builders consume selected candidates plus explicit split-group keys, produce typed counts/warnings/rejections, and attach split/group provenance for index finalization.
  - Index finalization consumes selected and split candidates, then emits ordered lazy `IndexItem`s with stable index-entry identity/fingerprints outside the Stage 3 `IndexItem` descriptor.
  - `DataSourceIndex` exposes length, item access, iteration, metadata, fingerprint, and optional entry/provenance access; existing `SampleBuilder` continues to consume each returned `IndexItem` one at a time.
  - `DataSourceIndexCodec` writes/reads schema-versioned JSON/JSONL-like manifests that wrap descriptor dicts, preserve resource URI/protocol/storage options, and reject unsupported schema versions or ambiguous fingerprints.
- Dependency direction:
  - `rphys.datasources.*` may depend on `rphys.data` vocabulary and `rphys.io` descriptors. `rphys.datasources.indexes` may import `IndexItem`, `RecordRef`, `FieldView`, `FieldLocator`, split/group metadata vocabulary, and primitive freezing helpers.
  - `rphys.io.codecs` remains datasource-neutral and must not import Stage 5 datasource builders/manifests. `SampleBuilder` remains one-item runtime construction and must not grow scan/split/index behavior.
  - Stage 5 must not depend on `rphys.ops`, export/save operations, Torch/cache/training, prediction/evaluation/analysis, dataset SDKs, numpy/pandas/scipy/torch/OpenCV/PyAV, `loom`, or workflow/artifact runtime packages.
- Extension points:
  - Datasource adapters and filters should be structural/importable Python objects; downstream config can reference them by `_target_` without editing `rphys` internals.
  - Use one reusable structural filter/result protocol over typed target units, not one untyped universal filter object. Concrete target records stay owned by their layer: pre-index datasource/view/record/field targets in filters, and `IndexCandidate` plus candidate selection results in indexes.
  - Builders should consume explicit plan objects and source objects. Plans/results should be primitive-or-descriptor serializable where useful, but not a generic stage runtime.
  - Symbolic adapter aliases, plugin discovery, public fake datasets, prepared-data cache manifests, and broad registry systems remain deferred unless a maintainer explicitly locks them.
- Compatibility constraints:
  - Preserve Stage 3/4 constructor and serialization semantics. Do not add scan/filter/build/manifest/fingerprint fields or methods to `DataSourceRef`, `RecordRef`, `DataSourceSchema`, `ResourceRef`, `FieldRef`, `FieldView`, or `IndexItem`.
  - Preserve existing package import-lightweight tests. Add Stage 5 submodules to lightweight import checks only when implemented and still dependency-free.
  - Keep `IndexItem` payload-free, transform-free, role-qualified, and compatible with `SampleBuilder`.
  - Candidate-level multiple-group/split policy is locked by DQ-5, sidecar index-entry identity with item-yielding access is locked by DQ-6, manifest envelope/fingerprint policy is locked by DQ-7, and source-aware composite-only combined-index semantics are locked by DQ-8.
