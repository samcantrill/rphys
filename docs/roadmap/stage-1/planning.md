## Roadmap Extraction

- Baseline roadmap outcome: implement the shared naming and error vocabulary used by every later `rphys` layer.
- Prerequisites: Stage 0 is assumed complete. Existing prerequisite evidence includes importable package homes, broad `RemotePhys*Error` classes, lightweight import tests, no runtime dependencies, private rights-status metadata, and README governance handoff.
- Prior or adjacent roadmap links: Milestone 0 created the skeleton and broad error baseline; Milestone 2 will consume these names in runtime `FieldSpec`, `FieldValue`, `Sample`, and `Batch`; later IO, datasource, operation, model, loss, metric, training, evaluation, and analysis milestones must not duplicate naming or error vocabulary.
- Primary feature docs: none found; `docs/roadmap.md` is the canonical source for this stage.
- Deferred or out-of-scope roadmap work: runtime field containers, lazy IO references, codecs, datasources, operations, transforms, models, losses, objectives, metrics, learners, trainers, prediction, evaluation, analysis, generic workflow runtime, artifact runtime, and raw dataset fixtures.
- Compatibility obligations: keep root and lightweight imports cheap; keep public imports intentional and tested; preserve `DataKey` identity separately from runtime role; reject role-prefixed paths and metadata selectors in `DataKey`; ensure `FieldLocator` preserves role, `DataKey`, and optional metadata key separately; ensure schema names are not conflated with data keys or codec keys; keep split names as partition/usage metadata rather than loop mode.
- Public surfaces or durable artifacts likely affected: `rphys.errors`; new modules under `rphys.data.keys`, `rphys.data.locators`, `rphys.data.schemas`, `rphys.data.types`, `rphys.data.metadata`, and `rphys.data.splits`; package, unit, and contract tests; README or roadmap-facing documentation once contracts are implemented.

## Overview

- What this stage covers: validated naming primitives, locator parsing/formatting, schema-name validation, metadata and split constants, and typed semantic diagnostics built on the Stage 0 broad error hierarchy.
- Why it exists: all later layers need one shared vocabulary before they can represent fields, lazy references, runtime containers, operations, metrics, or analysis outputs without inventing local naming conventions.
- Primary outcomes: deterministic validators; small public value types or string subclasses where appropriate; canonical constants for reserved namespaces, roles, data types, metadata keys, and splits; typed errors with inspectable context; tests that lock invalid and valid examples.
- Non-goals: data loading, runtime `Sample`/`Batch` behavior, collation, datasource indexing, codec resolution, transforms, model/loss/metric semantics, learner/trainer loop modes, registries, root package re-exports, config systems, workflow stages, artifact references, and optional scientific dependencies.
- Related feature docs: none found.
- Impacted repo areas: `src/rphys/errors.py`; `src/rphys/data/`; `src/rphys/data/__init__.py` if public exports are approved; `tests/unit/rphys/data/` or source-mirrored equivalent; `tests/unit/rphys/test_errors.py`; `tests/package/test_import.py`; `tests/package/test_import_boundaries.py`; `README.md` and `docs/roadmap.md` only if Stage 1 public behavior needs user-facing clarification.
- Current state: broad base errors exist and preserve message/context; `rphys.data` exists as an empty package home; no Stage 1 naming modules or semantic subclasses exist; package tests assert lightweight imports and empty planned namespace `__all__` surfaces.
- Key uncertainty: no blocking public-surface uncertainty remains after design approval; remaining details are implementation-level choices for exact diagnostic prose/context keys and approved strict-grammar edge cases.
- User clarification questions and resolved answers: approved 2026-05-12 / R2. Target audience is downstream `rphys` implementers/users and later roadmap stages. User-visible outcome is importable, validated vocabulary primitives/constants with typed diagnostics. Planning priority is a compact, strict, lightweight vocabulary layer with no runtime containers, registries, root re-exports, optional dependencies, datasource/IO/ops/model behavior, or workflow/artifact scope.
- Planning priority or optimization target: compact, strict, lightweight vocabulary layer that later milestones can depend on without refactoring, while avoiding registries, downstream config concerns, runtime container behavior, optional dependencies, and root package re-exports by default.

## Behavior Baseline

- Included behavior: deterministic validation and constants for `DataKey`, `FieldRole`, `FieldLocator`, `SchemaName`, `DataType`, `MetadataKey`, `SplitName`, and Stage 1 semantic errors.
- Default behavior: strict lowercase ASCII grammar, stdlib-only implementation, documented submodule imports, no root re-exports by default, no registries, no runtime dependencies, and no workflow/artifact scope.
- Failure behavior: invalid public vocabulary raises typed `RemotePhys*Error` subclasses with readable messages and inspectable context; validators fail loudly rather than silently normalizing ambiguous values.
- Unsupported behavior: any attempt to use Stage 1 names as loaders, runtime containers, datasource records, operation routes, codec selectors, loop modes, serialized manifest schemas, registries, or workflow artifacts is outside the approved behavior proposal.
- Resume/interruption behavior: record approvals and defaults in this document before moving gates.
- Downstream implications: Stage 1 becomes the canonical vocabulary dependency for runtime core, lazy IO, datasources, operations, methods, losses, metrics, training, evaluation, and analysis.
- Explicit deferrals: runtime containers, lazy IO, datasource behavior, operation/model/loss/metric/training behavior, serialization/config integration, registries, workflow/artifact runtime, root convenience re-exports, and optional dependencies.

  - `tests/contracts/`: narrow public vocabulary contract coverage for valid/invalid examples, locator round trips, typed diagnostics, and no runtime/workflow/artifact coupling.
- Likely public classes/functions/protocols:
  - `DataKey(str)`, `SchemaName(str)`, `MetadataKey(str)`, `SplitName(str)`, and `DataType(str)` as validated immutable string subclasses where downstream extension is expected.
  - `FieldRole(StrEnum)` as a closed set for the initial runtime roles from the roadmap.
  - `FieldLocator` as a frozen, slotted value object with `role: FieldRole`, `key: DataKey`, `metadata_key: MetadataKey | None`, a `parse()` constructor for `<role>/<data-key>[#<metadata-key>]`, and `__str__()` canonical formatting.
  - No public protocols, registries, serialization methods, manifest adapters, config hooks, runtime containers, or root package re-exports in Stage 1.
- Likely internal helpers:
  - Private component validators for lowercase ASCII tokens, dotted names, reserved namespace checks, and error construction.
  - Private constants for separators such as `.`, `/`, and `#`.
  - Private parser helpers returning primitive pieces; public classes own final coercion and typed errors.
- Data flow:
  - User strings enter through a vocabulary constructor or `FieldLocator.parse()`.
  - Validators reject unsupported values without case folding, trimming, aliasing, or silent normalization.
  - Valid string subclasses store the exact canonical string; `FieldLocator` stores validated components and formats back to the canonical locator string.
  - Locator parsing delegates `data-key` and metadata selector validation to `DataKey` and `MetadataKey` instead of duplicating those grammars.
  - Failures raise Stage 1 semantic errors with context such as `key`, `role`, `locator`, `schema`, `metadata_key`, `split`, `data_type`, `expected`, and `actual`.
- Dependency direction:
  - `rphys.errors` depends only on the standard library.
  - `rphys.data.keys`, `schemas`, `types`, `metadata`, and `splits` depend only on `rphys.errors` and private validation helpers.
  - `rphys.data.locators` may depend on `rphys.data.keys`, `rphys.data.metadata`, `rphys.errors`, and private validation helpers.
  - `rphys.data.__init__` remains empty by default and does not import submodules.
  - Deferred packages must not import back into `rphys.data` solely to define duplicate vocabularies.
- Extension points:
  - Downstream projects extend intrinsic field names through `custom.<project>.<semantic>[.<qualifier>...]`.
  - Downstream projects extend metadata and split labels by constructing validated `MetadataKey` and `SplitName` values.
  - Runtime roles remain the approved closed role vocabulary unless a later roadmap gate expands them.
  - Data type categories remain validated strings/constants, not a registry or backend type system.
- Compatibility constraints: Stage 1 grammars, error names, and module import paths become public contracts once approved, implemented, documented, and tested.
