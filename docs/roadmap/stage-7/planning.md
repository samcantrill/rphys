## Roadmap Extraction

- Baseline roadmap outcome: implement composable runtime operations over `Sample`
  and `Batch` objects, including `SampleOperation` and provisional
  `BatchOperation` families, transforms, augmentations, checks, and specialized
  pipelines. Roadmap `SampleOp`/`BatchOp` wording is treated as shorthand in
  this artifact.
- Prerequisites: Stage 2 runtime `Sample`/`Batch`/`FieldValue`/collation;
  Stage 3 lazy refs, `FieldView`, and `IndexItem`; Stage 4 `SampleField` and
  `SampleBuilder`; Stage 5 datasource/index construction; Stage 6 generic
  operation contracts, result-return execution, and ordered generic pipelines.
  Current code has these prerequisite surfaces present, and Stage 6 is now
  code-backed in `src/rphys/ops`.
- Prior or adjacent roadmap links: Stage 6 supplies generic operations and
  functional-kernel placement; Stage 8 will express save/export as normal
  operation flows; Stage 9 will let data loading optionally apply
  `SampleOperationPipeline`s and later `BatchOperation`s; Stage 10+
  methods/learners consume collated fields and self-supervised views without
  redefining operation semantics.
- Primary feature docs: none found.
- Deferred or out-of-scope roadmap work: concrete CHROM/POS/model-specific
  kernels, export/save/derived datasource writing, datasource scanning,
  DataLoader construction, cache/materialization plans, trainer loops, model
  input formatting, workflow/artifact runtime behavior, broad `BatchProgram`
  or graph language, top-level `rphys.transforms`, multi-member `IndexItem`s,
  and nested view `Sample`s.
- Compatibility obligations: preserve lightweight imports; use parsed
  `FieldLocator`s without changing locator meaning; enforce declared
  write/delete/dynamic permissions for `SampleOperation`s; avoid payload loading
  until an operation accesses payload data; keep default runtime mutation
  explicit and copy behavior explicit; require context-provided RNG streams and
  replayable seed material for augmentations; keep `BatchOperation`s provisional
  and equivalent to `SampleOperation` behavior when they replace it; keep
  numerical kernels separate from runtime containers and hidden device movement.
- Public surfaces or durable artifacts likely affected: likely new
  `rphys.ops.sample`, `rphys.ops.batch`, and `rphys.ops.pipelines` modules;
  package-level `rphys.ops` re-exports for implemented full names such as
  `SampleOperation`, `SampleTransform`, `SampleAugmentation`,
  `SampleOperationPipeline`, `SampleOperationContext`, `BatchOperation`,
  `BatchTransform`, `BatchAugmentation`, `BatchOperationPipeline`, and
  `BatchOperationContext`; possible concrete operation/pipeline errors in
  `src/rphys/errors.py`; unit tests under `tests/unit/rphys/ops/`; public
  contract tests under `tests/contracts/`; synthetic integration tests for lazy
  loading, pipeline order, augmentation replay, and SampleOperation/BatchOperation
  equivalence. No durable file format is directly required by Stage 7.

## Overview

- What this stage covers: locator-aware `SampleOperation` contracts, runtime
  transforms, deterministic checks, stochastic augmentations with replayable
  parameters, specialized sample operation pipelines, provisional
  `BatchOperation` contracts, BatchOperation equivalence rules, batch
  augmentation policy, and the initial self-supervised view-writing path.
- Why it exists: downstream preprocessing, augmentation, self-supervised view
  construction, runtime quality checks, and training data paths need operation
  contracts that understand `Sample`, `Batch`, `FieldLocator`, lazy fields,
  mutation permissions, RNG provenance, and pipeline ordering.
- Primary outcomes: SampleOperations validate declared input/output/deletion
  permissions and fail on undeclared mutation; `SampleOperationPipeline`
  preserves sequence and ordered-mapping order; SampleChecks are deterministic;
  sample and batch augmentations use explicit replayable parameter sampling;
  BatchOperations are allowed only as provisional optimization contracts with
  equivalence and provenance obligations.
- Non-goals: real rPPG algorithms or preprocessing catalogs, export/save ops,
  DataLoader adapters, cache/materialization, trainer/device movement policy,
  model formatting, datasource filters, index construction, workflow runtime
  abstractions, broad graph routing, hidden global registries, and raw datasets.
- Related feature docs: none found.
- Impacted repo areas: `src/rphys/ops/`; `src/rphys/errors.py`; package import
  tests; `tests/unit/rphys/ops/`; `tests/contracts/`; likely
  `tests/integration/`; public docstrings and possibly `docs/GLOSSARY.md` only
  if code-backed wording needs alignment.
- Current state: `Sample`, `Batch`, `FieldLocator`, `SampleField`, collation
  behavior, and Stage 6 generic operation contracts/pipelines are implemented.
  Stage 7-specific `SampleOperation`/`BatchOperation` families are not yet
  present in source.
- Key uncertainty: how Stage 7 public specialization classes should adapt the
  Stage 6 wrapper-first `Operation` design; how broad the initial
  BatchOperation surface should be; how explicit pipeline step names should
  become; how to model routing/drop decisions without drifting into DataLoader
  or workflow behavior; and how much RNG stream shape belongs in Stage 7 versus
  downstream framework adapters.
- User clarification questions and resolved answers: maintainer clarified on
  2026-05-14 that code-facing public names should prefer full
  `SampleOperation*` and `BatchOperation*` forms over abbreviated `SampleOp` and
  `BatchOp`, and that transform, augmentation, check, and related classes should
  inherit from the full base operation class when that remains the recommended
  code design.
- Planning priority or optimization target: establish the smallest
  field-permission-aware runtime operation layer that is scientifically
  inspectable, replayable where stochastic, compatible with lazy fields, and
  safe for later export, loading, caching, and training work.

## Behavior Confirmation

- Included behavior: Stage 7 builds the full-name runtime operation families
  over code-backed Stage 6 operations: `SampleOperation`,
  `SampleTransform`, `SampleAugmentation`, `SampleCheck`,
  `SampleOperationPipeline`, `SampleOperationContext`, provisional
  `BatchOperation`, `BatchTransform`, `BatchAugmentation`,
  `BatchOperationPipeline`, and `BatchOperationContext` where design confirms
  each public name. SampleOperations declare parsed `FieldLocator` reads,
  writes, deletes, and dynamic-field permissions; transforms write declared
  outputs; checks are deterministic; augmentations sample replayable params
  from explicit context seed/RNG material; pipelines preserve sequence and
  ordered-mapping order; BatchOperations carry provisional equivalence and
  provenance obligations when replacing SampleOperations; self-supervised view
  construction uses a wide-window sample and declared view locators.
- Default behavior: SampleOperations may mutate the input `Sample` in place and
  return it; non-mutating behavior copies explicitly. Read declarations support
  preflight/provenance, while field-set additions/removals/renames are checked
  against declared writes/deletes/dynamic permissions through before/after
  inventories or operation-owned helpers chosen during design. Pipeline and
  operation execution continue to expose `OperationResult` outputs/provenance
  rather than raw-output-only behavior.
- Failure behavior: missing or malformed locators, missing required fields,
  undeclared writes/deletes, invalid dynamic fields, invalid replay parameters,
  disallowed global/default RNG use, invalid route/check outputs, incompatible
  pipeline steps, and unsupported BatchOperation equivalence claims fail loudly
  with typed errors. Payload-internal mutation that does not change the field
  inventory is not transparently detected unless a later design adds explicit
  instrumentation.
- Unsupported behavior: concrete CHROM/POS/model-specific algorithm catalogs,
  export/save/derived datasource writing, datasource scanning or filtering,
  DataLoader construction, cache/materialization manifests, trainer/device
  movement policy, model tuple formatting, workflow/artifact runtimes,
  DAG/retry/resume pipeline languages, broad `BatchProgram`, multi-member
  `IndexItem`s, nested view `Sample`s, hidden registries, heavy optional base
  imports, and direct global/default NumPy/Torch/random RNG use.
- Resume/interruption behavior: this planning workflow remains docs-only; runtime resume/retry behavior is not introduced in Stage 7.
- Downstream implications: Stage 8 can wrap operation outputs for export/save
  without Stage 7 writing files; Stage 9 can apply SampleOperationPipelines in
  loaders and reason about deterministic materialization without Stage 7 owning
  cache keys; learners/methods can consume collated view fields without
  redefining augmentation or operation semantics.
- Explicit deferrals: exact inheritance versus wrapper adaptation, concrete
  context/parameter/equivalence record shapes, ordered mapping/name
  normalization, route result schema, concrete error class names, and precise
  module factoring are design-gate decisions. Concrete algorithms, export,
  cache/materialization, DataLoaders, trainers, workflow runtimes, model
  formatting, and broad batch programs are later-stage or downstream work.
- Why this behavior is locked: FQ-0 through FQ-8 are dependency ordered and
  resolved from maintainer naming guidance, `docs/roadmap.md`, code-backed
  Stage 6 operations, existing runtime container behavior, lazy-field
  contracts, and future-roadmap boundaries. No functionality queue item remains
  `needs maintainer discussion` or `blocked`.

## Proposed Implementation Shape

- Likely modules or packages:
  - `rphys.ops.sample` owns `SampleOperation`, `SampleTransform`,
    `SampleAugmentation`, `SampleCheck`, `SampleOperationContext`, and
    sample-specific contract/permission/result records needed to make those
    names real.
  - `rphys.ops.batch` owns provisional `BatchOperation`, `BatchTransform`,
    `BatchAugmentation`, `BatchOperationContext`, and batch-specific
    declaration/equivalence records.
  - `rphys.ops.core` and `rphys.ops.pipelines` may receive a broad foundation
    refactor before Stage 7 sample/batch work: introduce the operation-step
    execution interface, keep `Operation` as the standard callable wrapper,
    and make `OperationPipeline` depend on operation-step behavior rather than
    concrete `Operation` inheritance while preserving Stage 6 construction
    rules.
  - `rphys.ops.pipelines` keeps the generic `OperationPipeline`
    sequence-only behavior intact and adds specialized
    `SampleOperationPipeline` and `BatchOperationPipeline` classes for mapping
    diagnostics.
  - Private helpers can live in focused modules such as
    `rphys.ops._sample_validation`, `rphys.ops._field_permissions`, and
    `rphys.ops._pipeline_steps` if implementation needs them. They should not
    be imported from public docs/tests.
- Likely public classes/functions/protocols:
  - Full locked public operation names:
    `SampleOperation`, `SampleTransform`, `SampleAugmentation`,
    `SampleCheck`, `SampleOperationPipeline`, `SampleOperationContext`,
    `BatchOperation`, `BatchTransform`, `BatchAugmentation`,
    `BatchOperationPipeline`, and `BatchOperationContext`.
  - Proposed additional public records, subject to design agreement:
    `SampleOperationContract`, `SampleFieldPermissions`,
    `SampleAugmentationParams`, `SampleReplayRecord`, `SampleDecision`,
    `SampleRoute`, `BatchOperationContract`, `BatchAugmentationParams`,
    and `BatchEquivalenceReport`.
  - New public foundation interface:
    `OperationStep`, a small execution protocol/interface with `name`,
    `contract`, `run(input_value, context) -> OperationResult`, and callable
    semantics matching `run()`.
  - No registry is proposed. Downstream extension should remain through
    importable callables/classes wrapped by operation objects for ordinary
    users, with direct `OperationStep` implementation reserved for advanced
    adapters that need custom execution classes.
- Likely operation adaptation:
  - `Operation` should be the default concrete implementation of
    `OperationStep` for generic callables and continue to normalize
    `OperationResult` output.
  - `SampleOperation` and `BatchOperation` should implement `OperationStep`
    directly or by reusing `Operation` internally; they no longer need to
    inherit from concrete `Operation` for pipelines to accept them.
  - Specialized operation records still adapt to a generic
    `OperationContract(input_type=Sample, output_type=Sample)` or
    `OperationContract(input_type=Batch, output_type=Batch)` for compatibility
    checks, but specialized contracts remain separate and richer.
  - `run()` and `__call__()` should still return `OperationResult`; the
    specialized result details should live in result metadata/provenance or in
    typed supplemental records carried by the specialized contract/result.
  - `SampleTransform`, `SampleAugmentation`, and `SampleCheck` should inherit
    from `SampleOperation`; `BatchTransform` and `BatchAugmentation` should
    inherit from `BatchOperation`.
- Likely internal helpers:
  - Locator normalization from `FieldLocator | str` to immutable tuples of
    parsed `FieldLocator`s.
  - Permission snapshots based on `Sample.field_items()` and
    `Batch.field_items()` before and after execution.
  - Validation helpers for required read preflight, declared writes/deletes,
    dynamic-field predicates or role/key prefixes, and typed mutation errors.
  - Operation-step normalization that validates structural `OperationStep`
    behavior without accepting raw callables as pipeline steps.
  - Pipeline step normalization that accepts ordered sequences in generic
    pipelines, ordered mappings only in specialized pipelines, and renders an
    immutable step tuple with index and effective name.
  - RNG seed-material and replay-record validation using only Python standard
    library data structures and caller-provided RNG-like streams or seed
    objects.
- Data flow:
  - Generic pipeline construction normalizes `OperationStep` entries, rejects
    mappings and tuple named entries as Stage 6 did, and validates adjacent
    compatibility through each step's generic `OperationContract`.
  - Specialized pipeline construction normalizes steps and validates each entry
    is the correct specialized operation family.
  - Pipeline execution creates or accepts the specialized context, runs each
    step, forwards `OperationResult.output`, records step diagnostics, and wraps
    failures with step index plus effective step name.
  - A `SampleOperation` preflights declared reads without materializing payloads,
    chooses in-place or explicit copy mode, captures a field inventory, executes
    the wrapped callable, normalizes to `OperationResult`, captures the final
    inventory, and validates field-set mutations against declarations.
  - `SampleAugmentation.sample_params()` derives replayable params from
    `SampleOperationContext` seed/RNG material; `apply_params()` is deterministic
    for a fixed sample and params and writes only declared locators.
  - `BatchOperation` follows the same operation/result envelope over existing
    `Batch` fields, with explicit declaration/reporting for dtype/device,
    parameter scope, and equivalence to sample-level reference behavior.
- Dependency direction:
  - `rphys.ops.sample` may import `rphys.ops` Stage 6 primitives,
    `rphys.data.containers.Sample`, and `rphys.data.locators.FieldLocator`.
  - `rphys.ops.batch` may import Stage 6 primitives,
    `rphys.data.containers.Batch`, and `FieldLocator`; it must not import
    torch, numpy, DataLoader adapters, model code, datasource builders, cache,
    or export modules.
  - `rphys.ops.pipelines` may import only the operation-step interface and
    Stage 6 primitives for generic pipeline behavior; sample/batch-specific
    validation must stay in specialized pipeline paths to avoid circular or
    heavy imports.
- Extension points:
  - Ordinary users extend behavior by supplying importable callables/classes and
    explicit contract records to `Operation`, `SampleTransform`,
    `SampleAugmentation`, `SampleCheck`, `BatchTransform`, or
    `BatchAugmentation`, not by registering symbolic names globally.
  - Advanced adapters may implement `OperationStep` directly when they need a
    custom operation object, but they must still return `OperationResult`,
    expose a compatible contract, and remain dependency-light.
  - Backend-specific RNG streams, tensor devices, collators, loaders, exporters,
    and materializers stay behind later adapters. Stage 7 only records the
    runtime operation declarations those adapters can inspect later.
- Compatibility constraints:
