## Overview

- What this stage covers: unified engine-neutral events/profilers/callbacks;
  setup/teardown lifecycle events; native and concrete optional Lightning
  engine profiling integration; checkpoint/resume descriptors and engine hooks;
  precision/compile/kernel policy records; data-path and system/resource
  profiling; BatchOperation optimization validation; experiment tiers and
  restart evidence; documentation and validation guardrails.
- Why it exists: earlier stages built the data, operation, training, and
  analysis substrates. Stage 15 makes bottlenecks visible across training,
  dataloading, decoding, device transfer, compute, callbacks, checkpointing,
  validation, IO, CPU/GPU, memory, disk, network, and teardown so the next
  optimization decision is evidence-backed.
- Primary outcomes: the same profiler/event/callback contract works for native
  execution, Lightning execution, and future engines; profiles combine
  timestamped events, scalar span durations, and resource time-series traces so
  users can explain whether bottlenecks are data path, IO/decode, queueing,
  CPU, GPU, memory, disk, network, transfer, forward/backward/objective,
  optimizer, callbacks/logging, checkpointing, or validation;
  restart/checkpoint evidence is inspectable without a workflow runtime.
- Non-goals: new public method-output/result families; engine-specific
  user-facing profiling APIs; generic job runner/scheduler/workflow runtime;
  automatic tuning; hard Lightning/torch dependency in base imports; concrete
  optimized storage backend; raw-data benchmarks; mandatory GPU/system
  profilers in default tests.
- Related feature docs: none found.
- Impacted repo areas: `src/rphys/training/events.py`,
  `profiling.py`, `results.py`, `backend.py`, `plan.py`, new or existing
  checkpoint/policy/lightning modules under `rphys.training`,
  `src/rphys/datasources/datapath.py`, batch operation tests, package tests,
  integration tests, optional acceptance tests, and docs/docstrings.
- Current state: data-path records exist but are descriptor-only; native
  profiling emits unavailable placeholder spans; events have no timeline
  alignment fields; resource usage is not captured as a time series; event
  phases do not cover setup/teardown; checkpoint behavior is only a primitive
  `checkpoint_id` in `TrainingResult`; Lightning is intentionally absent;
  precision/compile/kernel policies are absent.
- Key uncertainty: exact design of the engine-neutral lifecycle/callback and
  checkpoint hook APIs. The functionality direction is locked, but design must
  prevent these hooks from becoming a generic workflow runtime. The optional
  Lightning module may expose framework-native input APIs, while shared
  results/events/profiles/checkpoints remain rphys contracts.
- User clarification questions and resolved answers: FQ-15-1 resolved on
  2026-05-18 by maintainer feedback.
- Planning priority or optimization target: maximize valid experiment
  decisions per unit cost by measuring the whole training/data path before
  introducing storage, framework, or kernel optimizations.

## Behavior Confirmation

- Included behavior: engine-neutral events/sinks/callbacks and setup/teardown
  lifecycle hooks; robust primitive profiling and run-decision records; native
  CPU profiling/checkpoint hooks; concrete optional first-class Lightning API
  through the shared `Trainer`/`TrainingEngine` path plus LightningModule,
  DataModule, or dataloader entrypoints under `rphys.training.lightning`;
  precision/compile/kernel policy descriptors; checkpoint/resume records and
  explicit hooks; data-path profiling/benchmarks; BatchOperation optimization
  validation; experiment tier/restart evidence; import/security/validation
  guardrails.
- Default behavior: CPU-only synthetic profiles and benchmarks are available
  without optional dependencies, raw data, hardware, network, or machine-speed
  thresholds. Optional probes and Lightning support are import-gated and record
  unavailable/unsupported evidence.
- Failure behavior: invalid records, policies, checkpoints, lifecycle events,
  and benchmark inputs raise typed errors. Callback/sink failures are
  fail-loud by default unless design explicitly introduces an evidence-backed
  skip/continue policy. Known unsafe optional dependency versions must be
  rejected or reported clearly before adapter execution.
- Unsupported behavior: separate engine-specific public plan/result/profile
  APIs; generic workflow/job/stage runner; cost dashboard; artifact store;
  hidden checkpoint serialization; automatic optimization/tuning; concrete
  optimized storage backend by default; mandatory GPU/system/Lightning checks
  in default CI; raw-data performance benchmarks.
- Resume/interruption behavior: Stage 15 records checkpoint refs, completion
  markers, loader/materialization fingerprints, restart tokens, and engine
  resume inputs. It does not own an end-to-end workflow resume scheduler or
  generic artifact lifecycle.
- Downstream implications: downstream projects and `loom` can drive job
  orchestration using typed profiles, lifecycle events, tier specs, checkpoint
  refs, and restart snapshots without `rphys` owning the workflow runtime.
- Explicit deferrals: concrete optimized storage backend, distributed
  orchestration, real data acceptance benchmarks, performance dashboards,
  automatic tuning, and public workflow/runtime abstractions.
- Why this behavior is locked: maintainer feedback on 2026-05-18 asked for
  seamless native/Lightning/future-engine profiling, concrete Lightning
  integration, precision/compile policy consideration, checkpoint/resume
  handling, setup/teardown hooks, and robust whole-path bottleneck diagnosis.

## Proposed Implementation Shape

Specialist-proposed design proposal on 2026-05-18, based on the approved
functionality/behavior baseline and pressure-tested against current
training/data-path code. The prior manager draft is revised here where the
evidence supports narrower boundaries.

- `rphys.training.events`: extend the existing `TrainingEvent` contract rather
  than adding a second lifecycle event family. Keep `TrainingEvent.phase` as the
  coarse vocabulary and keep start/completion/failure in `TrainingEvent.status`
  plus primitive metadata. Recommended additive phases: `setup`, `teardown`,
  `epoch`, `data_wait`, `device_transfer`,
  `forward`, `objective_or_loss`, `backward`, `optimizer_step`,
  `scheduler_step`, `checkpoint`, `validation`, `profile_summary`, and the
  existing loop/step/external phases. Use reserved metadata keys such as
  `stage_name`, `span_name`, `resource`, `engine`, `backend`, and
  `probe_name` only as primitive evidence. Add primitive timestamp/alignment
  fields or equivalent timeline metadata so events can be correlated with
  resource samples: at minimum a monotonic timestamp, a timeline identifier, and
  explicit clock/source notes, with optional wall-clock/process evidence where
  useful for external monitors. Keep sinks record-only; keep callbacks as
  side-effect hooks through `on_event(event)`; keep callback return values
  ignored; preserve fail-loud observer failures.
- `rphys.training.profiling`: keep `ProfileSpanSummary` as the durable span
  summary and add richer primitive records around it: `TrainingStepProfile`,
  `ResourceSample`, a grouped `ResourceTrace` or equivalent resource time-series
  record, `RunDecisionMetrics`, `TrainingProfile`, and a lightweight
  `TrainingProfileRecorder` or equivalent helper. `TrainingStepProfile` should
  carry named span summaries and counters for the whole step, not backend
  tensors or framework timeline objects. Scalar span summaries are appropriate
  for code-path durations such as data wait, forward, backward, optimizer,
  callback, checkpoint, and validation. Resource behavior such as CPU load,
  memory, GPU utilization, GPU memory, disk, network, and device transfer
  pressure must be represented as timestamped samples grouped into traces, not
  collapsed to one value per step. Resource samples should carry resource kind,
  device/process identity where known, value/unit, availability, sample time,
  sampling interval or sequence, overhead, synchronization notes, and clock
  alignment metadata. The recorder is an engine implementation helper, not a
  scheduler, logger, profiler backend, or artifact writer. Optional probes
  return `ResourceSample`/`ResourceTrace` evidence or
  `UnavailableProfileProbe` with explicit overhead and synchronization
  metadata.
- Resource monitoring should be an engine-neutral companion to span profiling.
  Native and Lightning execution should both be able to run the same rphys
  recorder/resource monitor, including optional background-thread or
  background-process samplers for device/system metrics when available. The
  core contract is the primitive trace output and timeline alignment evidence,
  not the sampling mechanism. Default tests use fake samplers; optional real
  probes may use system/GPU/framework libraries lazily and must report
  unavailable status, dropped samples, sampling cadence, and synchronization
  caveats rather than hiding them.
- A separate monitoring process is a valid optional sampler mode for GPU and
  host metrics because it can continue sampling while the training process is
  blocked in framework kernels, dataloader waits, checkpoint IO, Python GC, or
  native extensions. It should be lifecycle-bound to one training/profile run:
  start before the measured setup region, stop after teardown/profile flush,
  communicate through a bounded primitive channel or temporary trace file, emit
  monitor start/stop/failure events, and record clock-alignment, backpressure,
  dropped-sample, and orphan-cleanup evidence. It must not become a daemon,
  scheduler, artifact store, or required dependency path.
- `rphys.training.results`: preserve existing `TrainingResult.profiles` as a
  compatibility summary. Add one optional provisional `training_profile`
  aggregate field with a stable rphys-facing `TrainingProfile` inspection
  interface for detailed span/resource-trace/event/decision evidence.
  Avoid attaching raw profiler timelines, framework trainer state, optimizer
  state, checkpoint payloads, or callback internals. Lightning profiler or
  callback evidence may populate this field only after normalization into
  rphys profile records.
- `rphys.training.checkpoint` or an equivalent design-approved module: add
  `CheckpointPolicy`, `CheckpointRef`, `CheckpointResult`,
  `TrainingCheckpointer`, and `RestartState`. Native checkpoint behavior is
  hook/protocol based: rphys records identity, cadence, timing, status,
  failure, and restart compatibility, but the caller-owned hook writes and
  restores backend state. `CheckpointRef` is not a generic artifact reference;
  it should carry primitive identity/location/fingerprint/status evidence only.
  Lightning maps the same records to Lightning checkpoint callbacks and
  `ckpt_path`.
- `rphys.training.policies` or an equivalent design-approved module: add
  `PrecisionPolicy`, `CompilePolicy`, and `KernelPolicy` as descriptive,
  backend-applicable records. Native records unsupported/unavailable where it
  cannot enforce a policy. Lightning maps supported values to `Trainer`
  keyword arguments or module setup; unsupported values produce explicit
  diagnostics. Do not add a generic `engine_config` bag to `TrainingPlan`.
- `rphys.training.lightning`: add an optional module with
  first-class public Lightning support. Do not import it from
  `rphys.training.__init__` unless package import tests prove no heavy
  dependency loads. The adapter must perform a package/version preflight before
  importing an installed Lightning package when possible, block or clearly
  reject known compromised versions, then lazily import Lightning/torch. Stage
  15 should support both the shared `TrainingEngine`/`Trainer` path through an
  internal LightningModule adapter around an rphys `Learner` and explicit
  Lightning-native public entrypoints for user-authored LightningModule plus
  DataModule, dataloaders, callbacks, profiler, trainer kwargs, and `ckpt_path`
  execution. The optional Lightning API may use Lightning-specific input
  objects, but it must normalize back to rphys `TrainingResult`,
  `TrainingProfile`, `TrainingEvent`, checkpoint, and policy records. Lightning
  callback/profiler hooks should mark rphys spans/events and may feed normalized
  span evidence into the recorder, while resource monitoring should use the
  same engine-neutral rphys resource trace contract as native execution. Full
  distributed strategy parity and private Lightning state exposure remain out
  of scope for this implementation plan.
- `rphys.datasources.datapath`: do not reimplement Stage 9 descriptors. Add
  producer helpers that populate `DataPathProfile` and `DataPathBenchmark`
  using the same source/cache/prepared/materialized/collation path used by
  training. Specific timing keys that do not justify top-level fields should
  live in `summaries`/`measurements` with reserved names and documented units.
  If implementation proves those mappings too vague for validation, escalate a
  small typed data-path stage profile record rather than expanding the Stage 9
  descriptor family by default.
- `rphys.ops.batch` and tests: validate optimized `BatchOperation` behavior
  through equivalence reports, replay records, masks, dtype/device metadata,
  alignment/provenance checks, and failure cases before adding production
  optimization logic.
- Dependency direction: core training records stay standard-library and
  import-light. Training core must not import `rphys.datasources`, Lightning,
  torch, CUDA, psutil, NVML, pandas, numpy, plotting, logging frameworks, or
  dataset SDKs. Optional adapter/probe modules own those imports lazily.
  Datasource profiling producers may import datasource contracts but training
  core only consumes their primitive evidence.
- Extension points: structural event sinks, callbacks, profilers/resource
  probes, native checkpointers, framework adapters, future storage benchmark
  producers, and downstream/`loom` orchestration over typed evidence.

## Metric Taxonomy And Blind-Spot Audit

Maintainer refinement on 2026-05-18: Stage 15 should not treat
`CPU/GPU/disk/network` as an undifferentiated bucket. The shared
`TrainingProfile` schema should support a broad metric vocabulary through
primitive `ResourceTrace`/`ResourceSample` records, while adapters/probes record
`unavailable`, `unsupported`, or `ambiguous_attribution` evidence when a signal
cannot be collected safely or portably. Default validation remains fake-sampler
and threshold-free; optional real probes reduce risk where dependencies and
hardware are present.

| Area | Signals to represent when available | Why it matters | Attribution required | Blind spot if omitted | Default expectation |
| --- | --- | --- | --- | --- | --- |
| Timeline and event alignment | monotonic event timestamps, timeline ID, clock source, wall-clock/process evidence when useful, step/epoch/batch/rank/worker markers | Resource traces only become actionable when aligned with code phases. | engine, mode, timeline, process, rank/worker if present | Cannot correlate GPU/CPU spikes with data wait, transfer, backward, checkpoint, or validation. | Required with fake clocks. |
| Monitor execution mode | in-process, background-thread, or background-process sampler mode; monitor start/stop/failure events; queue/file backpressure; dropped samples; orphan cleanup; clock alignment handshake | GPU/system sampling may need to continue while the training process is blocked or busy. | monitor process ID where known, parent process, timeline, channel, sampler, lifecycle event | A thread-only sampler can miss stalls caused by Python/native blocking; an unmanaged process can leak or produce unaligned traces. | Fake process/thread samplers required; real process sampler optional. |
| Code-path spans | setup, data wait, decode/load, collate, device transfer, forward, objective/loss, backward, optimizer, scheduler, validation, callbacks/logging, checkpoint, teardown | Scalar duration summaries are the right shape for bounded code regions. | engine, mode, stage/span, step/epoch/batch, rank/worker where known | Bottleneck categories collapse into a single step duration. | Required with fake timers. |
| Accelerator utilization | GPU/device utilization, memory used/free/peak, allocation/reserved memory where framework exposes it, memory bandwidth or SM/tensor-core/compute activity where available, power, temperature, clocks/throttling, ECC/error or reset status when available | GPU utilization alone is too coarse; memory pressure, throttling, bandwidth, and allocation behavior can explain stalls. | host, process, rank, device ID, backend, sampler, units, cadence | Looks compute-bound or idle without distinguishing memory, transfer, thermal, or attribution issues. | Optional real probe; fake traces required. |
| Device transfer and synchronization | host-to-device/device-to-host bytes, transfer duration, pinned/pageable memory notes, async/sync status, stream or synchronization caveats, CPU-GPU synchronization markers | Transfers and hidden sync often dominate small batches. | stage/span, device, process/rank, transfer direction, sync note | Misattributes transfer stalls to model compute or data loading. | Required spans; optional byte/time traces. |
| Host CPU and process runtime | process and system CPU percent, per-core load where available, thread/process count, context switches, run queue/load average, Python runtime/GC pauses where available | High CPU or scheduling pressure can starve data loading or GPU submission. | process, worker, rank, host, sampler cadence | Parent-process-only CPU metrics hide dataloader worker saturation or contention. | Fake traces required; real probe optional. |
| Host memory and paging | RSS/USS/VMS where available, allocation high-water marks, swap, page faults, pinned/shared memory pressure, file cache notes where available | Memory pressure causes stalls, worker crashes, paging, and transfer issues. | process/worker/rank, host, units, sampler | GPU memory may look fine while host memory or pinned memory is the real limit. | Fake traces required; real probe optional. |
| Storage and filesystem IO | bytes read/written, throughput, operation counts/IOPS where available, latency summaries, open-file pressure, fsync/flush/checkpoint write time, cache hit/miss evidence | Slow storage, flushes, and cache misses often appear as data wait or checkpoint stalls. | path/URI kind when safe, source/cache/prepared/materialized stage, process/worker | IO bottlenecks get hidden inside dataloader wait or checkpoint spans. | Required synthetic data-path evidence; real device metrics optional. |
| Network and remote IO | bytes sent/received, request latency, retries/timeouts, remote URI/cache status, DNS/TLS/setup timing when available | Remote datasets, logs, and checkpoint uploads can dominate wall time. | URI scheme or redacted endpoint kind, process/worker, stage | Network stalls look like generic IO or callback/checkpoint time. | Optional; unavailable evidence required when absent. |
| Data-loader and queue behavior | worker count, worker lifecycle/failures, queue wait, queue depth or empty/full indicators, prefetch latency, batch assembly/collate time, decode time, cache hit/miss, records/sec/bytes/sec | Training can be GPU-idle because the input queue is empty. | worker, source/cache/prepared/materialized path, batch index, split | Cannot tell whether data wait is decode, collation, worker starvation, cache, or queueing. | Required synthetic data-path coverage. |
| Framework/autograd/compiler | graph compile time, graph breaks/fallbacks, warmup/cold-start markers, kernel/operation fallback notes, mixed-precision status, optimizer/scaler behavior, framework profiler spans after normalization | Compile and precision behavior can dominate first steps and change numerical meaning. | backend/framework, policy, mode, step, device | Steady-state and warmup costs are mixed; policy fallback is invisible. | Required fake mapping; real framework optional. |
| Distributed and multi-device | rank/world size, local/global rank, barrier/wait, all-reduce/communication spans, straggler evidence, per-rank resource traces, device/rank mapping | Distributed bottlenecks are often synchronization or one slow rank. | host, rank, world, device, strategy | Aggregated metrics hide stragglers and communication waits. | Optional/diagnostic; schema must support attribution. |
| Checkpoint/logging/reporting | serialization time, write/upload/flush duration, checkpoint size/identity/fingerprint/status, logger/callback/report spans, async/blocking notes | Checkpoint and reporting overhead can dominate or perturb training. | checkpoint policy/ref, destination kind, callback/logger name, rank | Runtime stalls are blamed on training compute instead of persistence/logging. | Required fake checkpoint spans/events. |
| Probe health and overhead | sampler start/stop, cadence, dropped samples, permissions, unavailable/unsupported reason, sampler overhead, synchronization caveats, ambiguity flags | Bad profiler data is worse than missing profiler data. | probe name, dependency, backend, process, clock source | Users overtrust noisy, low-resolution, or system-wide samples. | Required fake/unavailable coverage. |

Planned blind-spot controls:

- Metric records must be open enough for optional probe-specific metrics, but
  each sample still needs primitive `kind`, `value`, `unit`, `timestamp`,
  `resource`, `scope`, `availability`, and provenance rather than opaque payloads.
- Resource traces must support host/process/rank/worker/device attribution and
  an explicit `ambiguous_attribution` or equivalent status when only
  system-wide values are available.
- Separate monitor processes must be lifecycle-owned by the profile run, with
  explicit start/stop/failure events, bounded buffering, cleanup evidence, and
  no long-lived daemon or scheduler semantics.
- The implementation plan should include examples that show GPU idle due to
  data queue starvation, checkpoint write stalls, memory pressure, hidden
  transfer/synchronization cost, and unavailable optional probes.
- Real accelerator counters such as occupancy, memory bandwidth, tensor-core
  activity, power, thermals, and clocks are optional because portability varies;
  the schema must represent them when available and record absence clearly when
  not.
