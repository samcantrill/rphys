## Roadmap Extraction

- Baseline roadmap outcome: implement the in-memory runtime contract used after payloads are loaded or wrapped and before operations, methods, losses, metrics, and analysis consume fields.
- Prerequisites: Stage 1 naming, locators, schemas, metadata, splits, and typed validator errors are assumed complete per the maintainer instruction for this session.
- Prior or adjacent roadmap links: Stage 1 supplies `DataKey`, `FieldLocator`, `SchemaName`, `MetadataKey`, `DataType`, and split vocabulary. Stage 3 will add lazy IO references and index items and should consume Stage 2 runtime containers through builders rather than move IO behavior into `rphys.data`.
- Primary feature docs: none found; `docs/roadmap.md` is the canonical source.
- Deferred or out-of-scope roadmap work: lazy IO refs, codecs, datasource discovery, datasource scanning, transforms, operations, methods, models, losses, objectives, metrics, training loops, prediction/evaluation/analysis routines, serialization, manifests, and workflow/artifact runtime.
- Compatibility obligations: preserve lightweight imports, keep runtime role separate from `DataKey`, avoid hidden full-load or serialization behavior, fail loudly for missing fields and unsupported collation, and keep public imports intentional and tested.
- Public surfaces or durable artifacts likely affected: `rphys.data` package exports, new data runtime modules, `rphys.errors`, package tests, unit tests under `tests/unit/rphys/data`, narrow contract tests, and docstrings.

## Overview

- What this stage covers: `FieldSpec`, `FieldValue`, `DataObjectBase`, `CompositeDataObjectBase`, mutable `Sample` and `Batch`, public `FieldRequirement`, explicit `SampleContract`, `CollatePolicy.LIST`, `CollateContext`, and `collate_samples`.
- Why it exists: later operations, models, losses, metrics, and analysis need one loaded runtime representation before scientific behavior can be implemented without container-specific duplication.

- Non-goals: IO references, payload loading, codecs, datasource scanning, sample building from lazy refs, transforms, model execution, objective/metric math, training, persistence, config integration, and arbitrary payload serialization.
- Related feature docs: none found.

- Key uncertainty: no product/scientific blocker is apparent from the roadmap, but Stage 2 makes public API decisions for module layout, `rphys.data` exports, container mutation semantics, accessor returns, contract declaration shape, and initial collation behavior. ## Behavior Baseline

- Included behavior: minimal field specs; field values with narrow metadata; backend-free data-object hooks; mutable samples and batches; explicit sample contracts; LIST-only collation; runtime/collation typed errors; dependency-free public imports.
- Default behavior: constructors coerce Stage 1 vocabulary objects; containers wrap raw payloads in `FieldValue`; field access raises typed errors by default; collation requires explicit `CollatePolicy.LIST`.
- Failure behavior: invalid locators, missing fields, wrong payload types, schema mismatches, absent collation policies, unsupported policies, inconsistent field sets, and empty collation inputs fail loudly with typed diagnostics and context.
- Unsupported behavior: serialization, dict/JSON round trips, shape/dtype/unit/sample-rate validation, automatic tensor stacking, padding, allow-missing, custom collation callables, lazy loading, codecs, datasource scanning, operation execution, and training/evaluation logic.
- Resume/interruption behavior: no workflow/runtime resume behavior is introduced; copy methods are ordinary runtime object operations only.
- Downstream implications: later IO/sample builders can produce `Sample`; later operations can validate via `SampleContract`; later methods can consume `Batch`; richer scientific contracts can extend through specialized specs or data objects without changing basic containers.
- Explicit deferrals: tensor backend modules, rich field specs, schema catalogs, IO refs, datasource records, transforms, method/model/loss/metric semantics, persistence, and workflow/artifact integration.

  - `src/rphys/data/contracts.py`

- Likely internal helpers: private coercion helpers for locators/field values/policies and a private shared field-container base if it reduces duplication between `Sample` and `Batch`.
- Data flow: loaded payload -> `FieldValue` -> `Sample` fields keyed by `FieldLocator` -> `SampleContract.validate(sample)` or `collate_samples(samples)` -> `Batch` with LIST-collated payloads.
- Dependency direction: Stage 2 imports Stage 1 vocabulary and `rphys.errors`; it must not import `rphys.io`, `rphys.datasources`, `rphys.ops`, optional scientific stacks, or downstream workflow tools.
- Extension points: subclass `DataObjectBase` for simple loaded payloads; subclass `CompositeDataObjectBase` for explicit local child-data-object composition; pass arbitrary payloads through `FieldValue`; later specialized specs or contracts can add scientific validation outside the base containers.
- Compatibility constraints: public import paths, class names, equality/copy semantics, error names, and collation failure behavior become compatibility surfaces.
